Principi fondamentali

GRASP Patterns

General Responsibility Assignment Software Patterns — 9 principi guida per assegnare responsabilità a classi e oggetti. Non sono pattern implementativi come il GoF: sono criteri di giudizio per prendere buone decisioni di design.

IE

Information Expert

Assegna la responsabilità alla classe che possiede le informazioni necessarie. Chi sa, fa. Nel gioco, Player calcola il proprio punteggio perché ha score e lives.

responsabilità
CR

Creator

B crea A se B contiene, aggrega o usa strettamente A. GameSession crea i Player perché li aggrega e li contiene per tutta la sessione.

creazione
LC

Low Coupling

Minimizza le dipendenze. GameController dipende dall'interfaccia IGameObserver, non dalle sue implementazioni concrete — cambiare la UI non tocca il controller.

accoppiamento
HC

High Cohesion

Ogni classe ha responsabilità focalizzate e correlate. Item sa solo come applicarsi a un Player. Non gestisce sessioni, punteggi o livelli.

coesione
CT

Controller

Gli eventi di sistema sono gestiti da un oggetto Controller che non è UI. GameController riceve move(), collectItem(), endGame() dall'interfaccia e orchestra la logica.

architettura
PL

Polymorphism

Usa il polimorfismo invece di if/switch per variazioni di comportamento. Tipi di Item diversi (HealthItem, WeaponItem) implementano tutti apply(player) in modo diverso.

OOP
PF

Pure Fabrication

Classe artificiale non del dominio per rispettare LC e HC. Un GameRepository o ScoreService non esiste nel dominio del gioco, ma risolve problemi di persistenza senza sporcare le classi di dominio.

astrazione
IN

Indirection

Introduce un intermediario per disaccoppiare. L'interfaccia IGameObserver è un livello di indirezione tra GameSystem e le sue viste — la base del pattern Observer.

disaccoppiamento
PV

Protected Variations

Identifica i punti di cambiamento e proteggi le parti stabili con interfacce. La logica di rendering può cambiare liberamente: IGameObserver garantisce che il core del gioco non sia mai impattato.

stabilità
✅ Nel progetto del gioco: Controller → GameController; Low Coupling → uso di interfacce Observer; Information Expert → Player gestisce il proprio stato; Creator → GameSession crea i Player.
GoF · Creazionale

Singleton

Garantisce una sola istanza di una classe con un punto di accesso globale controllato. Nel gioco, GameSystem è un Singleton: esiste un solo sistema di gioco attivo che gestisce livelli, regole e stato globale.

Il costruttore è privato: nessuno può creare istanze dall'esterno. L'unico accesso avviene tramite il metodo statico getInstance(), che crea l'istanza al primo invio e la restituisce sempre nelle chiamate successive.

Vantaggi
  • Istanza controllata
  • Accesso globale
  • Lazy initialization
  • Stato condiviso sicuro
Attenzione
  • Difficile da testare
  • Stato globale nascosto
  • Thread safety richiesta
  • Viola SRP (spesso)
⚠️ Nei sistemi moderni preferisci la Dependency Injection. Il Singleton va usato solo quando l'unicità è un requisito reale del dominio, non solo convenienza.
GameSystem.java 
// «Singleton» — Thread-safe (Holder idiom)
public class GameSystem {

  // Holder: creato solo quando richiesto
  private static class Holder {
    private static final GameSystem
      INSTANCE = new GameSystem();
  }

  private GameSystem() {
    // inizializzazione livelli, regole…
  }

  public static GameSystem getInstance() {
    return Holder.INSTANCE;
  }

  public boolean checkWin(Player p) {
    return p.getScore() >= winThreshold;
  }

  public void nextLevel() { ... }
}

// Utilizzo da GameController
GameSystem sys = GameSystem.getInstance();
boolean won = sys.checkWin(player);
GoF · Comportamentale

Observer Pattern

Definisce una dipendenza uno-a-molti: quando il Subject cambia stato, tutti gli Observer registrati vengono notificati automaticamente. Nel gioco: GameSystem è il Subject, ScoreUI e SoundManager sono Observer.

IGameObserver.java 
public interface IGameObserver {
  void onScoreChange(int newScore);
  void onLevelUp(int level);
  void onGameOver();
}

// ScoreUI implementa l'Observer
public class ScoreUI
    implements IGameObserver {

  public void onScoreChange(int s) {
    display.update("Score: " + s);
  }
  public void onLevelUp(int l) {
    display.showBanner("Level " + l);
  }
  public void onGameOver() {
    display.showGameOver();
  }
}
GameSystem_notify.java 
// Nel Subject: notifica tutti gli observer
private List<IGameObserver> observers
    = new ArrayList<>();

public void addObserver(IGameObserver o) {
  observers.add(o);
}

private void notifyScoreChange(int s) {
  for (IGameObserver o : observers) {
    o.onScoreChange(s);
  }
}

// Registrazione dal Controller
sys.addObserver(new ScoreUI());
sys.addObserver(new SoundManager());
✅ L'Observer è il bridge tra il core del gioco e qualsiasi vista o effetto. Aggiungere un AchievementTracker non richiede modificare GameSystem: basta implementare IGameObserver e registrarlo.
GoF · I più usati nel progetto

Altri Pattern GoF

I 23 pattern GoF si dividono in Creazionali, Strutturali e Comportamentali. Ecco i più frequenti in un progetto software come il gioco, con spiegazione e contesto d'uso.

Strategy
Comportamentale Usato nel gioco

Definisce una famiglia di algoritmi intercambiabili e li incapsula, rendendoli sostituibili a runtime. Nel gioco: la strategia di movimento del Player può variare — camminata normale, scivolata, volo — senza cambiare la classe Player.

MovementStrategy.java 
public interface MovementStrategy {
  Position move(Position current, Direction dir);
}

public class WalkStrategy implements MovementStrategy {
  public Position move(Position p, Direction d) {
    return p.step(d, 1); // 1 tile
  }
}
public class FlyStrategy implements MovementStrategy {
  public Position move(Position p, Direction d) {
    return p.step(d, 3); // 3 tiles
  }
}

// Player usa la strategia corrente
player.setMovement(new FlyStrategy());
player.move(Direction.UP); // vola di 3 tile
Usalo quando: hai più varianti di un algoritmo che devono essere selezionabili a runtime, o per eliminare if/switch basati sul tipo di comportamento.
Factory Method
Creazionale

Definisce un'interfaccia per creare oggetti, delegando alle sottoclassi la scelta del tipo concreto. Nel gioco: ItemFactory crea Item diversi (HealthPotion, Shield, Weapon) senza che il Controller conosca le classi concrete.

ItemFactory.java 
public abstract class ItemFactory {
  public abstract Item createItem(String type);
}

public class LevelItemFactory extends ItemFactory {
  public Item createItem(String type) {
    return switch (type) {
      case "health" -> new HealthPotion(+30);
      case "weapon" -> new Sword("IronSword");
      default      -> new BasicItem();
    };
  }
}
Usalo quando: il codice che crea oggetti non deve conoscere il tipo concreto, o quando vuoi che sottoclassi decidano cosa istanziare.
Decorator
Strutturale

Aggiunge responsabilità a un oggetto dinamicamente, come alternativa all'ereditarietà. Nel gioco: un Player può ricevere un'armatura (ArmoredPlayer) che riduce i danni subiti, senza modificare la classe Player originale.

Usalo quando: vuoi aggiungere funzionalità a oggetti singoli senza sottoclassi, o per combinare comportamenti in modo flessibile a runtime.
Command
Comportamentale

Incapsula una richiesta come oggetto, permettendo parametrizzazione, accodamento e undo/redo delle azioni. Nel gioco: ogni azione del player (MoveCommand, UseItemCommand) può essere registrata e annullata.

Usalo quando: hai bisogno di undo/redo, registrazione delle operazioni (log), o vuoi accodare azioni da eseguire in futuro.
State
Comportamentale Usato nel gioco

Permette a un oggetto di cambiare comportamento al cambiare del suo stato interno. Nel gioco: GameSession può essere in stato WAITING, RUNNING, PAUSED, ENDED — ogni stato risponde diversamente agli stessi metodi.

GameState.java 
public interface GameState {
  void onMove(GameSession ctx, Direction d);
  void onPause(GameSession ctx);
}

public class RunningState implements GameState {
  public void onMove(GameSession ctx, Direction d) {
    ctx.getPlayer().move(d); // movimento normale
  }
  public void onPause(GameSession ctx) {
    ctx.setState(new PausedState());
  }
}

public class PausedState implements GameState {
  public void onMove(GameSession ctx, Direction d) {
    // ignorato: il gioco è in pausa
  }
}
Usalo quando: un oggetto cambia comportamento drasticamente in base al suo stato e vuoi evitare if/switch giganti sullo stato corrente.
Buone pratiche

Tecniche ricorrenti di design

Oltre ai pattern formali, esistono principi e tecniche che ogni progettista applica continuamente per mantenere il codice sano, evolutivo e testabile.

📐 Questi non sono pattern GoF formali, ma regole pratiche nate dall'esperienza. In un progetto reale, li applicherai più spesso dei pattern stessi.
Tecnica Categoria Descrizione e applicazione nel progetto
Program to Interface principio OOP Dipendi da astrazioni, non da classi concrete. GameController dipende da IGameObserver, non da ScoreUI.
Favor Composition over Inheritance principio OOP Invece di ereditare comportamento, iniettalo come dipendenza. Player usa una MovementStrategy composta, non estende una classe base di movimento.
SOLID — Single Responsibility SOLID Ogni classe ha un solo motivo per cambiare. Item.apply() solo effetti; la persistenza è in un GameRepository separato.
SOLID — Open/Closed SOLID Aperto all'estensione, chiuso alla modifica. Aggiungere un nuovo tipo di Item non modifica GameSystem — si aggiunge una nuova classe che implementa Item.
SOLID — Liskov Substitution SOLID Le sottoclassi devono essere usabili al posto della classe base senza rompere il comportamento. Ogni MovementStrategy concreta è intercambiabile.
SOLID — Interface Segregation SOLID Interfacce piccole e focalizzate. IGameObserver ha solo 3 metodi specifici — non un'unica interfaccia gigante che tutti devono implementare.
SOLID — Dependency Inversion SOLID I moduli di alto livello non dipendono da quelli di basso livello. GameController (alto livello) dipende da IGameObserver (astrazione), non da ScoreUI (basso livello).
Dependency Injection tecnica Le dipendenze vengono passate dall'esterno (costruttore, setter) invece di essere create internamente. Rende il codice testabile con mock e sostituibile senza modifiche.
Law of Demeter principio Parla solo con i tuoi amici diretti. Evita player.getInventory().getItem(0).apply(). Preferisci player.useFirstItem().
Tell, Don't Ask principio Dì agli oggetti cosa fare invece di chiedere il loro stato e decidere tu. Invece di controllare if player.lives == 0 → player.die(), chiama direttamente player.takeDamage() che gestisce internamente la morte.
Fail Fast tecnica Valida i precondizioni all'inizio dei metodi e lancia eccezioni immediate. Meglio un'eccezione chiara in collectItem(null) che un NullPointerException remoto e difficile da tracciare.
Immutability tecnica Oggetti di valore (Position, Score) immutabili eliminano bug di stato condiviso. Position.move(dir) restituisce una nuova Position invece di modificare quella esistente.
✅ Regola d'oro: se devi spiegare ad alta voce perché una classe fa quella cosa, probabilmente sta violando High Cohesion o Single Responsibility. Il codice ben progettato si legge come prosa.
