GRASP Patterns

General Responsibility Assignment Software Patterns — 9 principi guida per assegnare responsabilità a classi e oggetti in modo coerente e manutenibile.

IC

Information Expert

Assegna la responsabilità alla classe che possiede le informazioni necessarie per svolgere il compito. Chi sa, fa.

responsabilità
CR

Creator

La classe B dovrebbe creare istanze di A se B contiene, aggrega, registra o usa strettamente oggetti di tipo A.

creazione
LC

Low Coupling

Minimizza le dipendenze tra classi per ridurre l'impatto delle modifiche e aumentare la riusabilità del codice.

accoppiamento
HC

High Cohesion

Ogni classe dovrebbe avere responsabilità fortemente correlate e focalizzate. Evita le "God Class" che fanno tutto.

coesione
CT

Controller

Delega la gestione degli eventi di sistema a una classe controller, separando la logica UI dalla business logic.

architettura
PV

Polymorphism

Usa il polimorfismo invece di condizionali (if/switch) quando il comportamento varia in base al tipo.

OOP
PI

Pure Fabrication

Crea una classe artificiale (non del dominio) per raggiungere Low Coupling e High Cohesion. Es: DAO, Repository.

astrazione
IN

Indirection

Introduci un intermediario tra due componenti per disaccoppiarle. La base di molti design pattern.

disaccoppiamento
PV

Protected Variations

Identifica i punti di variazione e instabilità, poi costruisci interfacce stabili attorno ad essi.

stabilità
💡 Regola pratica: Low Coupling e High Cohesion sono i due principi più importanti. Ogni altra decisione di design dovrebbe tendere a massimizzarli entrambi contemporaneamente.

Deep dive — Pattern Creazionale

Singleton

Il pattern più semplice e più controverso del GoF. Garantisce l'esistenza di una sola istanza di una classe nell'intera applicazione.

Come funziona

Il costruttore viene reso privato per impedire la creazione diretta di istanze. L'accesso avviene solo tramite un metodo statico getInstance() che, al primo invio, crea l'istanza e la memorizza in un attributo statico privato.

Tutte le chiamate successive restituiscono sempre la stessa istanza già creata, garantendo un unico punto di accesso globale.

Singleton.java 
// Implementazione Thread-Safe (Eager)
public class DatabaseConnection {

  // Istanza creata al caricamento della classe
  private static final DatabaseConnection
    instance = new DatabaseConnection();

  // Costruttore privato
  private DatabaseConnection() {
    // inizializzazione connessione
  }

  // Punto di accesso globale
  public static DatabaseConnection
    getInstance() {
      return instance;
  }

  public void query(String sql) {
    // esecuzione query
  }
}

// Utilizzo
DatabaseConnection db =
  DatabaseConnection.getInstance();
db.query("SELECT * FROM users");
Singleton_lazy.java 
// Double-Checked Locking (Lazy)
public class Singleton {

  private volatile static Singleton instance;

  private Singleton() {}

  public static Singleton getInstance() {
    if (instance == null) {
      synchronized (Singleton.class) {
        if (instance == null) {
          instance = new Singleton();
        }
      }
    }
    return instance;
  }
}
⚠️ Anti-pattern warning: Il Singleton è spesso considerato un anti-pattern in sistemi moderni. Preferire la Dependency Injection per mantenere il codice testabile e disaccoppiato. Il Singleton dovrebbe essere usato solo quando l'unicità dell'istanza è un requisito reale del dominio, non solo di convenienza.
