Build Tools (Strumenti di Costruzione)

Quando un progetto software cresce, compilare manualmente centinaia di file sorgenti, scaricare le librerie esterne e creare i pacchetti eseguibili diventa impossibile. I Build Tools automatizzano tutto questo processo.

Maven

Storico tool per l'ecosistema Java basato sul paradigma "Convention over Configuration".

  • pom.xml: File centrale (Project Object Model) che descrive le dipendenze da scaricare in automatico e i plugin da eseguire.
  • Repository: Maven scarica le librerie richieste da server centralizzati (Maven Central) in una cache locale sul computer dello sviluppatore.
  • Lifecycle: Offre fasi standard (es. compile, test, package, install) eseguibili in sequenza logica.
Gradle

Strumento più moderno e flessibile rispetto a Maven. Anziché usare lunghi file XML, utilizza script dichiarativi basati su Groovy o Kotlin. È il build tool ufficiale per lo sviluppo Android ed è noto per la sua velocità grazie ai meccanismi avanzati di caching incrementale.

Source Code Management (SCM)

Il controllo di versione (o versioning) è essenziale per permettere a team di programmatori di collaborare sullo stesso codice senza sovrascrivere il lavoro altrui e potendo recuperare versioni precedenti in caso di errori.

Centralizzato vs Distribuito
  • Centralizzato (es. SVN): C'è un solo server che possiede tutto lo storico del progetto. Per visualizzare le vecchie versioni o committare è necessaria una connessione di rete continua col server.
  • Distribuito (es. Git): Ogni sviluppatore ha una copia completa (clone) dell'intero repository sul proprio PC. Le operazioni avvengono in locale in modo ultraveloce, e la sincronizzazione con gli altri (push/pull) avviene solo quando necessario.
Concetti base di Git

Flusso standard di lavoro per chi usa Git:

  • Commit: Registra un'istantanea (snapshot) delle modifiche effettuate nella history locale.
  • Branch: Una ramificazione parallela. Permette di sviluppare una nuova funzionalità (feature branch) senza rompere il codice del ramo principale (main/master).
  • Merge: L'operazione che unisce le modifiche di un branch secondario all'interno del branch principale, risolvendo eventuali "conflitti".
  • Push & Pull: Rispettivamente, inviare i propri commit locali al server remoto (es. GitHub) o scaricare quelli altrui.

Automazione e Qualità (CI/CD)

La pratica DevOps mira a ridurre la barriera tra chi scrive il codice (Dev) e chi gestisce i server e le operazioni (Ops), automatizzando l'intero ciclo di rilascio.

Continuous Integration (CI)

Ogni volta che uno sviluppatore esegue un push del codice, un server automatico (es. Jenkins o GitHub Actions) compila il codice e lancia tutti i test per assicurarsi che non si sia rotto nulla. Questo rileva gli errori immediatamente.

Analisi Statica e SonarQube

Durante la pipeline automatica, strumenti come SonarQube analizzano il testo sorgente prima che venga eseguito per individuare Code Smells (cattivo design, alto accoppiamento), vulnerabilità di sicurezza e calcolare la percentuale di codice coperta dai test automatici (Code Coverage).

Virtualizzazione e Container (Docker)

Spesso un'app funziona sul PC di chi l'ha creata ma fallisce sui server di produzione perché mancano librerie o versioni specifiche di un database. I container risolvono questo problema ("Works on my machine!").

Cosa sono i Container

A differenza delle tradizionali Macchine Virtuali (VM) che emulano un intero hardware e installano un sistema operativo pesante per ogni applicazione, i Container (come Docker) condividono lo stesso nucleo del sistema ospitante, impacchettando solo il software e le librerie strettamente necessarie. Sono istantanei all'avvio e leggerissimi.

  • Immagine (Image): Il pacchetto immutabile che contiene il codice sorgente, il runtime e le dipendenze. È come uno stampo.
  • Container: Un'istanza in esecuzione di un'immagine.
