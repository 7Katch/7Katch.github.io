    GRASP · 01

    Information Expert

    Assegna la responsabilità alla classe che possiede le informazioni necessarie per adempierla. Il principio è semplice: chi sa, fa. Non spostare i dati fuori dalla classe per elaborarli altrove.

    IE

    Definizione formale

    Quando devi assegnare una responsabilità, inizia chiedendoti: chi ha i dati per soddisfarla? Quella classe è l'esperta dell'informazione e dovrebbe gestire la responsabilità in modo autonomo. Questo riduce l'accoppiamento, perché non è necessario esporre i dati verso l'esterno.
    responsabilità
    Esempio 1 — Gioco (dal progetto)

    Player ha score e lives. Quindi è Player a calcolare il proprio totale punti con getTotalScore(), non un manager esterno che legge i campi. Se il calcolo cambia, si modifica solo Player.

    Esempio 2 — E-commerce

    Un Order conosce i propri OrderItem. Quindi il totale dell'ordine si calcola con order.getTotal(), non con un servizio esterno che somma i prezzi dopo averli estratti uno a uno. Order è l'esperto del suo contenuto.

    Esempio 3 — Biblioteca

    Un oggetto Book conosce il proprio ISBN, titolo e autore. È lui a decidere se due libri sono uguali (equals()) e come si rappresenta come stringa (toString()). Non serve un BookComparator esterno che accede ai campi pubblici.

    Esempio 4 — Scacchi

    Un ChessPiece conosce la propria posizione e tipo. È lui a dire se una mossa è valida: piece.isValidMove(from, to). Non è il Board che deve conoscere le regole di ogni singolo pezzo.

    ❌ Violazione

    Un GameController legge player.score e player.multiplier e calcola il totale lui stesso. I dati stanno in Player ma la logica è altrove.

    ✅ Corretto

    player.getTotal() incapsula il calcolo dove stanno i dati. Il controller chiama un'unica operazione e non dipende dai dettagli interni.

    Player.java — Information Expert 
    public class Player {
  private int baseScore;
  private double multiplier;
  private int bonusPoints;

  // ✅ Player è l'esperto: conosce tutti i dati necessari
  public int getTotalScore() {
    return (int)(baseScore * multiplier) + bonusPoints;
  }

  // ✅ Solo Player sa se ha ancora vite
  public boolean isAlive() {
    return lives > 0;
  }
}

// ❌ Violazione — il controller non dovrebbe sapere come si calcola il punteggio
// int total = (int)(player.baseScore * player.multiplier) + player.bonusPoints;
    GRASP · 02

    Creator

    B dovrebbe creare istanze di A se: B contiene o aggrega A; B registra A; B usa strettamente A; o B ha i dati di inizializzazione di A. La regola evita creazioni sparse ovunque nel codice.

    CR

    Chi crea chi?

    La responsabilità di creare un oggetto appartiene a chi ha il contesto per farlo in modo sensato. Centralizzare la creazione riduce la dipendenza da dettagli costruttivi e rende il codice più testabile.
    creazione
    Esempio 1 — Gioco

    GameSession aggrega i Player per tutta la sessione, quindi è GameSession a crearli: new Player(name, startingLives). Ha anche i dati di configurazione iniziale.

    Esempio 2 — Fatturazione

    Un Invoice aggrega le sue InvoiceLine. È Invoice ad aggiungere righe: invoice.addLine(product, qty, price), non un servizio esterno che costruisce righe e le inietta.

    Esempio 3 — Forum

    Un Thread contiene Post. Quando un utente risponde, è il Thread a creare il nuovo Post: thread.addReply(author, content). Il Thread ha tutto il contesto: data, posizione nella sequenza, metadati.

    Esempio 4 — Gioco di carte

    Un Deck (mazzo) aggrega e registra tutte le Card. È lui a crearle nella fase di setup: deck.build() genera le 52 carte. Nessun'altra classe deve sapere come si costruisce una carta.

    ❌ Violazione

    Un GameController fa new Player(...) in mezzo alla logica di gioco. Il controller non aggrega Player, quindi non è il responsabile naturale della creazione.

    ✅ Corretto

    GameSession.startSession() crea i Player. La sessione li possiede, li gestisce, e ha tutti i parametri di setup necessari.

    Collegamento →  Creator e Factory Method sono complementari: quando la creazione diventa complessa o richiede varianti, si passa a una Factory senza violare Creator — la Factory diventa il creatore naturale.
    GRASP · 03

    Low Coupling

    Minimizza le dipendenze tra classi. Un'alta dipendenza (high coupling) significa che una modifica in A costringe a cambiare B, C, D. Con basso accoppiamento le classi sono intercambiabili e testabili in isolamento.

    LC

    Forme di accoppiamento

    L'accoppiamento può essere su tipi concreti (il peggiore), su interfacce (accettabile) o su astrazioni (ottimale). L'obiettivo non è zero dipendenze — impossibile — ma dipendenze verso le astrazioni giuste.
    accoppiamento
    Esempio 1 — Gioco

    GameController dipende da IGameObserver, non da ScoreUI o SoundManager. Cambiare tutta la UI non richiede toccare il controller.

    Esempio 2 — Repository pattern

    Un servizio dipende da IUserRepository, non da MySQLUserRepository. In test si inietta un InMemoryUserRepository; in produzione si usa quello reale. Il servizio non cambia mai.

    Esempio 3 — Notifiche

    Un OrderService non chiama direttamente EmailSender.send() e SmsSender.send(). Dipende da INotifier. Se si aggiunge WhatsApp, basta aggiungere un'implementazione — il servizio non si tocca.

    Esempio 4 — Logging

    Le classi di business non chiamano System.out.println() né dipendono da Log4j. Dipendono da ILogger. L'implementazione può essere sostituita (console, file, cloud) senza toccare la logica applicativa.

    ❌ High Coupling

    GameController importa e istanzia ScoreUI, SoundManager, AchievementTracker. Aggiungere un observer richiede modificare il controller.

    ✅ Low Coupling

    GameController ha una lista di IGameObserver. Si aggiungono observers a runtime senza toccare il controller. Testabile con mock.

    GRASP · 04

    High Cohesion

    Ogni classe dovrebbe avere responsabilità strettamente correlate e focalizzate. Una classe con bassa coesione fa troppe cose diverse ed è difficile da capire, mantenere e riusare.

    HC

    Test della coesione

    Prova a descrivere la classe in una frase senza usare "e". Se non ci riesci, probabilmente la classe ha responsabilità multiple e andrebbe spezzata. Una classe coesa fa una cosa sola e la fa bene.
    coesione
    Esempio 1 — Gioco

    Item sa solo come applicarsi a un Player: apply(player). Non gestisce sessioni, non salva nulla, non notifica nessuno. Una responsabilità.

    Esempio 2 — Classe coesa

    EmailValidator fa una cosa: validare indirizzi email. Non invia email, non le salva nel DB, non le formatta. Ogni metodo è correlato alla validazione.

    Esempio 3 — Da spezzare

    Una classe UserManager che: autentica utenti, manda email, aggiorna il profilo, calcola statistiche e scrive log — viola completamente High Cohesion. Va divisa in AuthService, ProfileService, NotificationService, UserStatsService.

    Esempio 4 — Valore immutabile

    Money (con importo e valuta) è altamente coesa: tutti i suoi metodi — somma, sottrai, converti, confronta — riguardano le operazioni su un valore monetario. Non fa nient'altro.

    ❌ Bassa coesione

    Una GameManager class che inizializza il gioco, gestisce l'input utente, salva lo stato, suona gli effetti audio e aggiorna la UI. Ogni parte è separata dalle altre.

    ✅ Alta coesione

    GameSession gestisce solo il ciclo di vita della partita. InputHandler solo l'input. AudioManager solo l'audio. Ciascuna ha responsabilità focalizzate.

    ⚠️ HC + LC vanno insieme: spezzare una classe può aumentare il coupling se non si usano interfacce. I due principi si bilanciano: alta coesione interna a ogni classe, basso accoppiamento tra le classi.
    GRASP · 05

    Controller

    Gli eventi di sistema (azioni dell'utente, richieste esterne) devono essere ricevuti da un oggetto Controller che non è parte dell'UI. Il Controller coordina la risposta senza fare la logica di business da solo.

    CT

    Due tipi di Controller

    Facade Controller: un unico oggetto per tutti gli eventi di sistema (es. GameController per tutti gli eventi del gioco). Adatto a sistemi più semplici.

    Use Case Controller: un controller per ogni caso d'uso (es. LoginController, CheckoutController). Più scalabile per applicazioni grandi.
    architettura
    Esempio 1 — Gioco

    GameController riceve move(direction), collectItem(itemId), endGame() dall'interfaccia e orchestra la logica: valida, aggiorna il modello, notifica gli observer. Non è UI.

    Esempio 2 — Web app (MVC)

    In un'app web, OrderController riceve la request HTTP e chiama orderService.placeOrder(dto). Non contiene logica di business — la delega al service layer. Non è una vista — restituisce dati al framework.

    Esempio 3 — ATM

    ATMController riceve gli eventi fisici (carta inserita, PIN digitato, importo selezionato) e coordina CardReader, PinValidator, CashDispenser. L'utente interagisce solo con lo schermo fisico, mai con questi oggetti direttamente.

    Esempio 4 — Sistema ospedaliero

    AppointmentController gestisce i casi d'uso "prenota visita" e "cancella appuntamento". Separa la UI (form web) dalla logica (controllo disponibilità, invio reminder) e dalla persistenza (repository).

    ❌ Sbagliato

    Un pulsante UI fa direttamente player.lives-- e scoreLabel.setText(...). La logica di gioco è nell'interfaccia grafica. Impossibile testare, riusare o cambiare la UI.

    ✅ Corretto

    Il pulsante chiama gameController.onPlayerHit(). Il controller aggiorna il modello, il modello notifica gli observer, la UI si aggiorna da sola. Separazione netta.

    GameController.java 
    public class GameController {

  private final GameSession    session;
  private final GameSystem     system;
  private final List<IGameObserver> observers;

  // Evento di sistema: l'utente si muove
  public void onMove(Direction dir) {
    session.getCurrentPlayer().move(dir);   // delega al modello
    notifyAll(Event.MOVE);                  // notifica la vista
  }

  // Evento di sistema: raccoglie un item
  public void onCollectItem(String itemId) {
    Item item = session.getItem(itemId);
    item.apply(session.getCurrentPlayer());  // Information Expert
    session.removeItem(itemId);
    notifyAll(Event.ITEM_COLLECTED);
  }
}
    GRASP · 06

    Polymorphism

    Quando il comportamento varia in base al tipo, usa il polimorfismo invece di if/switch. Ogni tipo gestisce la propria variazione implementando un'interfaccia comune.

    PL

    Polimorfismo vs. if/switch

    Un if (type == "HEALTH") ... else if (type == "WEAPON") ... deve essere aggiornato ogni volta che si aggiunge un tipo. Con il polimorfismo si aggiunge una nuova classe — il codice esistente non si tocca (Open/Closed Principle).
    OOP
    Esempio 1 — Gioco: tipi di Item

    HealthPotion, Sword, Shield implementano tutti Item.apply(player) in modo diverso. Il controller chiama item.apply(player) senza sapere il tipo concreto.

    Esempio 2 — Pagamenti

    CreditCard, PayPal, BankTransfer implementano IPaymentMethod.charge(amount). CheckoutService chiama solo method.charge(total), senza if/switch sul tipo di pagamento.

    Esempio 3 — Forme geometriche

    Circle, Rectangle, Triangle implementano Shape.area() e Shape.perimeter(). Un renderer chiama shape.draw() senza sapere quale forma sta disegnando.

    Esempio 4 — Report

    PdfReport, ExcelReport, HtmlReport implementano IReport.generate(data). Il servizio di esportazione non cambia quando si aggiunge un nuovo formato.

    ❌ Senza polimorfismo

    if (item.type.equals("health")) player.heal(30);
    else if (item.type.equals("weapon")) player.addWeapon(...);
    Ogni nuovo tipo richiede di toccare questo blocco.

    ✅ Con polimorfismo

    item.apply(player);
    Ogni implementazione sa cosa fare. Aggiungere ShieldItem non modifica niente di esistente.

    Item.java — Polymorphism 
    public interface Item {
  void apply(Player player);
  String getName();
}

public class HealthPotion implements Item {
  private final int healAmount;
  public void apply(Player p) { p.heal(healAmount); }
}

public class Sword implements Item {
  private final int damage;
  public void apply(Player p) { p.setAttack(damage); }
}

public class Shield implements Item {
  private final int defense;
  public void apply(Player p) { p.setDefense(defense); }
}

// ✅ Il controller non sa (e non deve sapere) qual è il tipo
for (Item item : collectedItems) {
  item.apply(currentPlayer);
}
    GRASP · 07

    Pure Fabrication

    Classe "artificiale" che non esiste nel dominio del problema, ma viene creata per rispettare Low Coupling e High Cohesion. Risolve problemi tecnici (persistenza, logging, comunicazione) senza sporcare le classi di dominio.

    PF

    Quando inventare una classe

    Quando nessuna classe del dominio è il posto giusto per una responsabilità tecnica (salvare su DB, inviare email, fare cache), si inventa una classe di servizio. Non deve avere senso nel dominio reale — deve avere senso nel design.
    astrazione
    Esempio 1 — Gioco

    ScoreService e GameRepository non esistono nel dominio del gioco, ma gestiscono persistenza e logica di punteggio senza inquinare Player o GameSession con dipendenze tecniche.

    Esempio 2 — E-commerce

    Un TaxCalculator non è un'entità del dominio ordini, ma è necessario per calcolare le tasse. Meglio una classe dedicata che mettere la logica fiscale dentro Order o Product.

    Esempio 3 — Social network

    FriendRecommendationEngine non è un concetto del dominio "amicizia", ma è una Pure Fabrication che incapsula l'algoritmo di raccomandazione. L'utente non sa che esiste, ma il sistema ne ha bisogno.

    Esempio 4 — Mapper

    Un UserMapper che converte tra UserEntity (DB) e UserDTO (API) è una Pure Fabrication. Non corrisponde a nulla nel dominio reale, ma risolve il problema del disaccoppiamento tra layer.

    ❌ Senza Pure Fabrication

    Player contiene metodi per salvarsi su DB, inviare notifiche email e calcolare statistiche. Classe gonfia, dipendenze tecniche nel dominio, impossibile testare in isolamento.

    ✅ Con Pure Fabrication

    Player è solo logica di dominio. PlayerRepository gestisce la persistenza. PlayerNotifier le notifiche. Ciascuno testabile, sostituibile, indipendente.

    GRASP · 08

    Indirection

    Introduce un intermediario per ridurre l'accoppiamento diretto tra due classi. L'intermediario assorbe le dipendenze e consente a ciascun lato di evolversi indipendentemente.

    IN

    Il valore dell'intermediario

    "All problems in computer science can be solved by another level of indirection." — David Wheeler. L'indirezione è la base di pattern come Observer, Adapter, Proxy, Facade, e dell'intera architettura a layer.
    disaccoppiamento
    Esempio 1 — Gioco

    IGameObserver è il livello di indirezione tra GameSystem e ScoreUI / SoundManager. I due lati non si conoscono, comunicano solo attraverso l'interfaccia.

    Esempio 2 — Adapter

    La tua app usa IEmailService. Oggi l'implementazione chiama SendGrid, domani Mailgun. L'adapter è l'intermediario che traduce l'interfaccia comune nell'API specifica del provider. L'app non cambia mai.

    Esempio 3 — Message Broker

    Un microservizio A pubblica un evento su Kafka. Il microservizio B lo consuma. Non si conoscono direttamente. Kafka è il livello di indirezione che li disaccoppia: A e B possono scalare, aggiornarsi e fallire indipendentemente.

    Esempio 4 — DNS

    Il DNS è indirezione pura: il client chiede "dove trovo google.com?" e riceve un IP. Se Google cambia server, aggiorna il DNS — tutti i client continuano a funzionare senza sapere nulla del cambio.

    ❌ Dipendenza diretta

    GameSystem chiama scoreUI.update(), soundManager.play(), achievementTracker.check() direttamente. Aggiungere un effetto richiede modificare GameSystem.

    ✅ Con indirezione

    GameSystem chiama notifyObservers(event). Chi vuole reagire implementa IGameObserver e si registra. GameSystem non conosce chi ascolta.

    Collegamento →  Indirection e Pure Fabrication spesso coincidono: l'intermediario che si introduce è spesso una Pure Fabrication (una classe che non esiste nel dominio, ma serve per disaccoppiare).
    GRASP · 09

    Protected Variations

    Identifica i punti del sistema dove le variazioni sono prevedibili (algoritmi, formati, provider esterni, regole di business) e proteggi le parti stabili con interfacce. Le variazioni sono libere; il nucleo stabile non si tocca mai.

    PV

    Cosa è stabile, cosa cambia?

    Il design migliore isola ciò che cambia da ciò che rimane stabile. Ciò che è stabile (interfaccia pubblica, contratto) va protetto. Ciò che cambia (implementazione, algoritmo, fornitore) va isolato dietro un'astrazione.
    stabilità
    Esempio 1 — Gioco

    Il rendering può cambiare (console, GUI Swing, web): IGameObserver protegge il core. La logica di gioco non cambia mai per colpa del rendering. Il contratto è stabile; l'implementazione è libera di variare.

    Esempio 2 — Sorting

    ISortStrategy protegge il codice che ordina dati dalle variazioni di algoritmo (quicksort, mergesort, timsort). Si può cambiare l'algoritmo a runtime senza toccare chi usa il sort.

    Esempio 3 — Provider di pagamento

    Il contratto IPaymentGateway è stabile. Le implementazioni (StripeGateway, PayPalGateway) possono cambiare o aggiungersi. Il sistema di checkout non sa quale gateway usa e non cambia.

    Esempio 4 — Serializzazione

    ISerializer con serialize(obj) e deserialize(bytes) protegge il codice dal formato: JSON, XML, Protobuf, MessagePack. Si cambia implementazione senza toccare la logica applicativa.

    Esempio 5 — Cambio di database

    L'interfaccia IProductRepository è stabile. Sotto c'è PostgreSQLProductRepository. Migrare a MongoDB significa scrivere MongoProductRepository senza toccare nessun servizio. Il contratto protegge tutto il resto.

    ❌ Non protetto

    Il codice di business chiama direttamente metodi specifici di Stripe. Cambiare provider significa riscrivere metà applicazione. Ogni variazione di API esterna si propaga nel codice interno.

    ✅ Protetto

    Il codice di business chiama IPaymentGateway.charge(amount). La variazione (quale provider) è isolata dietro l'interfaccia. Il business logic non cambia mai per motivi tecnici.

    Un colpo d'occhio su tutti i principi GRASP con la domanda chiave da porsi durante il design.

    GoF · Creazionale

    Singleton

    Garantisce una sola istanza di una classe con un punto di accesso globale controllato. Nel gioco, GameSystem è un Singleton: esiste un solo sistema di gioco attivo che gestisce livelli, regole e stato globale.

    Il costruttore è privato: nessuno può creare istanze dall'esterno. L'unico accesso avviene tramite il metodo statico getInstance(), che crea l'istanza al primo invio e la restituisce sempre nelle chiamate successive.

    Vantaggi
    • Istanza controllata
    • Accesso globale
    • Lazy initialization
    • Stato condiviso sicuro
    Attenzione
    • Difficile da testare
    • Stato globale nascosto
    • Thread safety richiesta
    • Viola SRP (spesso)
    ⚠️ Nei sistemi moderni preferisci la Dependency Injection. Il Singleton va usato solo quando l'unicità è un requisito reale del dominio, non solo convenienza.
    GameSystem.java 
    // «Singleton» — Thread-safe (Holder idiom)
public class GameSystem {

  // Holder: creato solo quando richiesto
  private static class Holder {
    private static final GameSystem
      INSTANCE = new GameSystem();
  }

  private GameSystem() {
    // inizializzazione livelli, regole…
  }

  public static GameSystem getInstance() {
    return Holder.INSTANCE;
  }

  public boolean checkWin(Player p) {
    return p.getScore() >= winThreshold;
  }

  public void nextLevel() { ... }
}

// Utilizzo da GameController
GameSystem sys = GameSystem.getInstance();
boolean won = sys.checkWin(player);
    GoF · Comportamentale

    Observer Pattern

    Definisce una dipendenza uno-a-molti: quando il Subject cambia stato, tutti gli Observer registrati vengono notificati automaticamente. Nel gioco: GameSystem è il Subject, ScoreUI e SoundManager sono Observer.

    ✅ Nel progetto del gioco: gli Observer vengono notificati quando il punteggio o lo stato del Player cambia. Questo garantisce il Low Coupling tra il core e la UI.
