Di cosa parliamo

01 Sistemi Operativi Base
12 argomenti ▼
Definizione fondamento
Il SO è il software che gestisce l'hardware e fornisce servizi alle applicazioni. Fa da intermediario tra l'utente e le risorse fisiche della macchina.
Struttura "Computer System"
Architettura a strati: hardware → kernel → sistema operativo → applicazioni → utente. Ogni livello espone servizi al livello superiore.
Sistemi Multiprocessori
Più CPU condividono memoria e bus. Aumentano throughput e affidabilità. Si distinguono in sistemi simmetrici (SMP) e asimmetrici.
Architettura SMP & Multicore design
Symmetric Multi-Processing: tutte le CPU hanno uguale accesso alla memoria. Il multicore integra più core nello stesso chip, riducendo latenza e consumo.
Non-Uniform Memory Access (NUMA)
In sistemi con molte CPU, ogni processore ha memoria "locale" (accesso rapido) e memoria "remota" (accesso più lento). Il SO deve minimizzare accessi remoti.
Servizi OS
Gestione processi, memoria, file system, I/O, comunicazione inter-processo, protezione e sicurezza, interfaccia utente (CLI/GUI).
System Calls
Interfaccia programmabile tra applicazione e kernel. Esempi: fork(), read(), write(), open(), exit(). Causano un cambio di contesto da user mode a kernel mode.
Implementazione delle System Calls
Tramite trap/interrupt software: l'applicazione carica un numero di syscall in un registro e genera un'interruzione. Il kernel legge il numero e chiama l'handler corretto.
Esempi di System Calls
Categorie: Process control (fork, exec), File management (open, read, write), Device management (ioctl), Communications (socket, send).
Struttura OS
Approcci architetturali: monolitico (Linux), a strati, a microkernel, modulare. Ogni approccio bilancia performance, manutenibilità e sicurezza.
Microkernels
Solo le funzioni essenziali nel kernel (IPC, gestione memoria base, scheduling). Tutto il resto (driver, filesystem) gira in user space come server. Più sicuro, ma overhead di comunicazione.
Moduli
I kernel moderni (Linux) supportano moduli caricabili dinamicamente: driver e funzionalità aggiuntive aggiunte/rimosse a runtime senza riavviare il sistema.
02 Processi Core
14 argomenti ▼
Concetti del processo fondamento
Un processo è un programma in esecuzione. Include il codice (text section), i dati (heap, stack) e le informazioni di contesto. Ogni processo ha il proprio spazio di indirizzi.
Process Control Block (PCB)
Struttura dati nel kernel che rappresenta un processo: PID, stato, program counter, registri CPU, informazioni di scheduling, gestione memoria, lista file aperti.
Stato del processo
Ciclo di vita: New → Ready → Running → Waiting → Terminated. Le transizioni sono gestite dallo scheduler e dagli eventi I/O.
Context Switch
Salvataggio del PCB del processo corrente e ripristino del PCB del prossimo processo da eseguire. È overhead puro: durante il switch la CPU non esegue lavoro utile.
Creazione dei processi
In Unix/Linux tramite fork(): crea un processo figlio identico al padre. Il figlio poi chiama exec() per caricare un nuovo programma. Il padre può attendere con wait().
Albero dei processi in Linux
Ogni processo (tranne PID 1 — init/systemd) ha un processo padre. Si forma una gerarchia ad albero ispezionabile con pstree.
Terminazione processo
Tramite exit() (normale) o segnali (SIGKILL, SIGSEGV). I processi terminati diventano zombie finché il padre non chiama wait() per raccogliere il valore di uscita.
IPC — Shared Memory
Due o più processi condividono una regione di memoria fisica. Comunicazione velocissima (nessuna copia), ma richiede sincronizzazione esplicita per evitare race condition.
IPC — Message Passing
I processi si scambiano messaggi tramite send/receive. Più sicuro della shared memory (nessuna memoria condivisa), ma overhead per la copia del messaggio nel kernel.
Implementazione del Communication Link
Il link può essere diretto (processi si nominano esplicitamente) o indiretto (tramite mailbox/porta). Può essere unidirezionale o bidirezionale, sincrono o asincrono.
Sincronizzazione dei messaggi
Bloccante (sincrona): il mittente aspetta che il ricevente riceva. Non bloccante (asincrona): il mittente continua dopo l'invio. La combinazione più comune: send non bloccante + receive bloccante.
Buffering
I messaggi sono accodati in un buffer prima della ricezione. Capacità zero (rendez-vous), limitata o illimitata. Determina il comportamento del blocco del mittente.
Pipes
Canale di comunicazione unidirezionale tra processi correlati (ordinarie) o non correlati (named pipe/FIFO). Implementa un buffer FIFO nel kernel. Usate intensamente nelle shell Unix (ls | grep foo).
THREADS anticipa cap.03
Unità di esecuzione all'interno di un processo. Condividono codice, dati e file aperti con gli altri thread dello stesso processo, ma hanno stack e registri propri.
03 Thread Core
9 argomenti ▼
Benefici del multithreading
Responsività (UI sempre reattiva), condivisione risorse (nessuna IPC necessaria), economia (creare thread costa meno che creare processi), scalabilità su sistemi multicore.
Concorrenza vs Parallelismo
Concorrenza: più task in progresso nello stesso tempo (anche su singola CPU, con interleaving). Parallelismo: task eseguiti letteralmente nello stesso istante su più core.
Parallelismo dei dati e dei task
Data parallelism: stesso calcolo su sottoinsieme di dati su ogni core (es. somma di array). Task parallelism: operazioni diverse su dati diversi su ogni core.
Legge di Amdahl
Speedup massimo = 1 / (S + (1-S)/N) dove S è la frazione seriale e N il numero di core. Anche con infiniti core, la parte seriale limita il guadagno. Mostra l'importanza di ridurre le sezioni sequenziali.
User e Kernel Threads
User threads: gestiti dalla libreria a livello utente (es. pthreads), invisibili al kernel. Kernel threads: gestiti direttamente dal kernel, possono essere schedulati su core diversi.
Modelli di multithreading
Definisce come i thread utente si mappano sui thread kernel. Determina parallelismo reale e overhead.
Many-to-One
Molti thread utente → 1 thread kernel. La libreria gestisce il multiplexing. Se un thread si blocca in I/O, tutti si bloccano. Nessun parallelismo reale. Usato in sistemi senza supporto kernel ai thread.
One-to-One
Ogni thread utente → 1 thread kernel. Parallelismo reale su multicore. Overhead maggiore per ogni creazione. Modello usato da Linux e Windows.
Many-to-Many
N thread utente → M thread kernel (M ≤ N). Compromesso: parallelismo reale + flessibilità. Il kernel crea thread sufficienti e la libreria li gestisce. Thread pool pattern.

04 CPU Scheduling Gestione CPU
8 argomenti ▼
Concetti base & Criteri fondamento
Lo scheduler decide quale processo/thread occupa la CPU. Criteri: CPU utilization (max), Throughput (max), Turnaround time (min), Waiting time (min), Response time (min).
FCFS — First Come First Served
Il primo processo nella ready queue viene eseguito fino al termine (non preemptive). Semplice ma soffre dell'effetto convoy: un processo lungo blocca tutti i brevi dietro di sé.
SJF — Shortest Job First
Esegue prima il processo con CPU burst più breve. Minimizza il waiting time medio, ma richiede previsione del burst (spesso tramite media esponenziale). Può causare starvation dei processi lunghi.
Round Robin (RR)
Ogni processo riceve un quanto di tempo fisso (time quantum, tipicamente 10-100ms). Alla scadenza viene preempted e rimesso in fondo alla coda. Ottimo per time-sharing, il quantum influenza il tradeoff context switch / response time.
Priority Scheduling
Ogni processo ha una priorità; la CPU va al processo a priorità più alta. Problema: starvation dei processi a bassa priorità. Soluzione: aging (la priorità aumenta col tempo di attesa).
Multilevel Queue
Più code separate per categorie di processi (foreground, background, batch). Ogni coda ha il proprio algoritmo e priorità. I processi non si spostano tra code.
Multilevel Feedback Queue
Come Multilevel Queue ma i processi possono muoversi tra code in base al comportamento. Un processo CPU-bound scende di priorità; uno interattivo sale. Il più flessibile e usato nella pratica.
CFS — Completely Fair Scheduler (Linux)
Lo scheduler di Linux moderno. Usa un albero rosso-nero ordinato per virtual runtime: esegue sempre il processo con meno vruntime. Garantisce equità e ottima interattività senza quantum fisso.
05 Sincronizzazione Concorrenza
8 argomenti ▼
Race Condition problema
Situazione in cui il risultato dipende dall'ordine di esecuzione di thread/processi concorrenti. Esempio classico: due thread incrementano la stessa variabile senza sincronizzazione → risultato non deterministico.
Sezione critica & Mutual Exclusion
Una sezione critica è codice che accede a risorse condivise. Tre requisiti: Mutua esclusione (un solo processo alla volta), Progress (decisione non indefinita), Bounded waiting (nessuna starvation).
Mutex Lock
Lock binario: acquire() prima della sezione critica, release() dopo. Se il lock è occupato, il thread attende (busy waiting con spinlock, o sospensione con mutex blocking).
Semafori
Variabile intera con due operazioni atomiche: wait(S) (P) decrementa e blocca se S≤0, signal(S) (V) incrementa e sveglia un thread in attesa. Binario (0/1) o contatore (gestisce N risorse).
Monitor
Costrutto di alto livello (Java: synchronized). Garantisce che un solo thread alla volta esegua un metodo del monitor. Usa condition variables per wait()/notify().
Problema Produttore-Consumatore
Il produttore genera dati, il consumatore li usa. Buffer condiviso di dimensione N. Soluzione con semafori: empty (slot liberi), full (slot pieni), mutex (accesso esclusivo al buffer).
Problema dei Lettori-Scrittori
Più lettori possono leggere simultaneamente; uno scrittore richiede accesso esclusivo. Varianti: priorità ai lettori (possibile starvation scrittori) o priorità agli scrittori.
Problema dei 5 Filosofi
Classico problema di sincronizzazione: 5 filosofi, 5 bacchette condivise. Ogni filosofo ne serve 2. Naive solution → deadlock. Soluzioni: asimmetria, pickup atomico, semaforo limite (max 4 filosofi a tavola).
06 Deadlocks Problemi
6 argomenti ▼
Condizioni necessarie (Coffman) fondamento
Tutte e 4 devono verificarsi simultaneamente: Mutual exclusion, Hold and wait (tieni e aspetti), No preemption (risorse non sottraibili), Circular wait (catena circolare di attesa).
Grafo di Allocazione delle Risorse
Grafo diretto: processi (cerchi) e risorse (rettangoli). Arco P→R = richiesta; arco R→P = assegnazione. Se il grafo ha un ciclo con risorse a istanza singola → deadlock. Con istanze multiple → possibile ma non certo.
Prevenzione del Deadlock
Elimina almeno una delle 4 condizioni Coffman. Esempio: imponi ordine globale sulle risorse (elimina circular wait). Oppure: richiedi tutte le risorse in anticipo (elimina hold-and-wait). Conservativa, riduce l'utilizzo delle risorse.
Algoritmo del Banchiere (Avoidance)
Dijkstra. Prima di concedere una risorsa, simula l'allocazione e verifica se il sistema rimane in uno stato sicuro (esiste una sequenza sicura di completamento). Richiede di conoscere le risorse massime di ogni processo.
Rilevamento del Deadlock
Permette al deadlock di avvenire, poi lo rileva periodicamente tramite un algoritmo simile al banchiere ma senza la dichiarazione massima. Overhead ridotto, ma serve recovery.
Recovery dal Deadlock
Process termination: termina tutti i processi nel deadlock, o uno alla volta finché il ciclo si rompe. Resource preemption: sottrai risorse a un processo scelto (con possibile rollback al checkpoint).

07 Main Memory Memoria
8 argomenti ▼
Allocazione contigua base
Ogni processo occupa un blocco contiguo di memoria fisica. Semplice ma soffre di frammentazione esterna. Strategie: first-fit, best-fit, worst-fit per scegliere il buco libero.
Frammentazione
Esterna: spazio totale libero sufficiente, ma non contiguo. Interna: la memoria allocata è leggermente più grande di quella richiesta (spreco all'interno del blocco). La paginazione elimina la frammentazione esterna.
Paginazione
La memoria fisica è divisa in frame di dimensione fissa; lo spazio logico in pagine della stessa dimensione. La page table mappa pagine logiche → frame fisici. Elimina la frammentazione esterna ma introduce overhead della tabella.
TLB — Translation Lookaside Buffer
Cache hardware delle traduzioni pagina→frame più recenti. Riduce drasticamente il costo della traduzione degli indirizzi. TLB hit: 1 ciclo. TLB miss: accesso alla page table in memoria (~100 cicli). Hit rate tipico: 99%+.
Protezione della memoria
Ogni entry della page table include bit di protezione: read/write/execute. Il tentativo di accesso non autorizzato genera una trap (segmentation fault). Isola i processi tra loro e dal kernel.
Pagine condivise
Più processi possono mappare lo stesso frame fisico. Usato per librerie condivise (libc), copy-on-write dopo fork(), e memoria condivisa IPC. Risparmio significativo di RAM.
Segmentazione
Lo spazio logico è diviso in segmenti di dimensione variabile (codice, stack, heap, dati). La segment table mappa ogni segmento con base + limite. Più naturale per il programmatore, ma soffre di frammentazione esterna.
Segmentazione con paginazione
Ibrido usato da x86: ogni segmento ha la propria page table. Combina la visione logica della segmentazione con l'assenza di frammentazione esterna della paginazione.
08 Virtual Memory Memoria
7 argomenti ▼
Demand Paging fondamento
Le pagine vengono caricate in memoria solo quando necessarie (lazy loading). Il valid bit nella page table indica se la pagina è in RAM. Se non lo è, si genera un page fault.
Page Fault
Trap generata quando si accede a una pagina non in RAM. Il SO localizza la pagina su disco, trova un frame libero (o lo libera), carica la pagina, aggiorna la page table, e riprende il processo. Costo: ~10ms vs ~100ns per accesso RAM.
Sostituzione delle Pagine
Quando non ci sono frame liberi, il SO deve scegliere una vittima. Algoritmi: FIFO, OPT (ottimale, non implementabile), LRU (Least Recently Used), Clock (approssimazione LRU usata in pratica).
Allocazione dei frame
Equal allocation: stessi frame per ogni processo. Proportional allocation: in base alla dimensione del processo. Priority allocation: processi con priorità alta ricevono più frame.
Thrashing
Il processo spende più tempo a gestire page fault che a eseguire codice utile. Accade quando i frame assegnati sono insufficienti per il working set del processo. Il SO deve rilevarlo e ridurre il grado di multiprogramming.
Working Set Model
Il working set è l'insieme di pagine accedute nelle ultime Δ referenze. Se la somma dei working set supera i frame disponibili → thrashing imminente. Il SO monitora i working set per allocare frame adeguati.
Memory-Mapped Files
Un file su disco viene mappato nello spazio di indirizzi virtuale del processo. L'accesso al file avviene tramite normali istruzioni di memoria. Il SO gestisce il caricamento lazy tramite demand paging. Base di mmap() su Unix.

09 I/O Systems Hardware
7 argomenti ▼
Hardware I/O fondamento
Porte, bus e device controller. Il controller ha registri di stato, controllo e dati. Il SO comunica con il controller tramite I/O port-mapped (istruzioni IN/OUT) o memory-mapped I/O (registri mappati in RAM).
Polling vs Interrupt-driven I/O
Polling: la CPU controlla periodicamente se il dispositivo è pronto (busy-wait, spreca cicli CPU). Interrupt: il dispositivo notifica la CPU quando ha finito. La CPU può fare altro nel frattempo.
DMA — Direct Memory Access
Il controller DMA trasferisce blocchi di dati tra dispositivo e RAM senza coinvolgere la CPU. La CPU inizia il trasferimento e viene interrotta solo al completamento. Essenziale per dischi, schede di rete, audio.
Kernel I/O Subsystem
Strato del kernel che astrae i dispositivi. Fornisce: scheduling I/O, buffering, caching, spooling (per dispositivi che accettano un job alla volta, es. stampanti), gestione degli errori.
Buffering I/O
Buffer in memoria per adattare velocità diverse tra produttore e consumatore, tra dati e dimensione dei trasferimenti, e per la semantica copy-on-write. Single buffer, double buffer, circular buffer.
Driver di dispositivo
Modulo del kernel che traduce le richieste generiche del kernel I/O subsystem in comandi specifici per il controller hardware. Fornisce un'interfaccia uniforme (open, read, write, ioctl) indipendente dall'hardware sottostante.
I/O sincrono vs asincrono
Sincrono: il processo si blocca fino al completamento dell'I/O. Asincrono: il processo continua; viene notificato al completamento (callback, signal, future/promise). Fondamentale per I/O ad alto throughput.
10 Mass-Storage Systems Storage
6 argomenti ▼
HDD vs SSD fondamento
HDD: piatti magnetici rotanti, testine mobili. Accesso sequenziale veloce, random lento (seek time + rotational latency). SSD: memoria flash NAND, nessun componente meccanico. Accesso random uniforme, bassa latenza, ma usura per scritture.
Disk Scheduling
Ottimizza l'ordine delle richieste I/O su HDD per minimizzare il movimento delle testine. Non necessario per SSD. Algoritmi: FCFS (semplice), SSTF (nearest first, rischio starvation), SCAN/C-SCAN (elevatore).
SCAN & C-SCAN
SCAN: la testina va avanti e indietro come un ascensore, servendo le richieste in direzione. C-SCAN: va solo in una direzione; al bordo salta all'inizio. Distribuzione più uniforme dei tempi di attesa.
RAID
Redundant Array of Independent Disks. Livelli: RAID 0 (striping, performance), RAID 1 (mirroring, affidabilità), RAID 5 (striping + parità distribuita, bilanciamento), RAID 6 (doppia parità, resiste a 2 guasti).
Storage Area Network (SAN)
Rete dedicata ad alta velocità per collegare server a storage centralizzato. I dischi appaiono come locali al server. Scalabile, gestito centralmente. Usato in datacenter e ambienti enterprise.
Gestione dello spazio libero
Strutture dati per tracciare i blocchi liberi: Bit vector (1 bit per blocco, compatto, facile accesso), Linked list (blocchi liberi collegati), Grouping e Counting (ottimizzazioni).
11 File System File
7 argomenti ▼
Concetti base: file e directory fondamento
Un file è una collezione di dati con un nome. La directory mappa nomi → file (o altre directory). Strutture: singolo livello, a due livelli, ad albero, con link (grafo aciclico o generale).
Allocazione dei file su disco
Contigua: blocchi consecutivi, accesso rapido, frammentazione esterna. Linked: blocchi collegati, nessuna frammentazione, accesso random lento. Indexed (inode): blocco indice con puntatori ai blocchi dati. Usato da Unix/Linux.
inode (Unix)
Struttura dati che descrive un file: permessi, proprietario, dimensione, timestamp, puntatori diretti (primi 12 blocchi), indiretto singolo, doppio e triplo. Il nome del file è nella directory, non nell'inode.
FAT — File Allocation Table
Tabella in memoria che mappa ogni cluster al successivo (linked list su FAT). FAT12/16/32 per dimensioni crescenti. Semplice, compatibile, ma senza journaling e permessi avanzati. Usato su USB e SD card.
Journaling
Il filesystem registra le modifiche in un log (journal) prima di applicarle. In caso di crash, il journal permette il recovery rapido senza fsck completo. Usato da ext4, NTFS, XFS, APFS.
VFS — Virtual File System
Strato di astrazione nel kernel che fornisce un'interfaccia uniforme (open, read, write) sopra filesystem diversi (ext4, NTFS, procfs, NFS). Le applicazioni non conoscono il filesystem sottostante.
Mounting & Link
Mounting: aggancia un filesystem a un punto dell'albero delle directory. Hard link: due nomi per lo stesso inode (stessa directory entry, stesso file). Soft link (symlink): file speciale con un percorso, può puntare tra filesystem diversi.

12 Embedded Systems Speciale
5 argomenti ▼
Sistemi Real-Time fondamento
Sistemi in cui la correttezza dipende sia dal risultato che dal tempo in cui viene prodotto. Esempi: controllo ABS, pacemaker, sistemi avionici. Il SO deve garantire deadline deterministiche.
Hard vs Soft Real-Time
Hard RT: una deadline mancata è un failure del sistema (airbag, controllo reattore). Soft RT: una deadline mancata degrada la qualità ma non è catastrofico (streaming video, UI).
Rate Monotonic Scheduling (RMS)
Algoritmo preemptive a priorità statica: i task con periodo più breve hanno priorità più alta. Ottimale per task periodici a priorità fissa. Limite di utilizzazione: ≈69% per n task (formula di Liu & Layland).
EDF — Earliest Deadline First
Priorità dinamica: il task con deadline più vicina ha priorità più alta. Ottimale per sistemi RT monoprocessore: può schedulare qualsiasi insieme di task schedulabile. Più complesso da implementare di RMS.
Caratteristiche RTOS
Un Real-Time OS ha: latenza di interrupt minimale e deterministica, preemption immediata, no virtual memory (latenza non deterministica), clock ad alta risoluzione. Esempi: FreeRTOS, VxWorks, QNX, Zephyr.
13 Sicurezza Protezione
6 argomenti ▼
Obiettivi di sicurezza fondamento
Confidenzialità: solo chi è autorizzato vede i dati. Integrità: i dati non vengono alterati non autorizzatamente. Disponibilità: il sistema è accessibile a chi ne ha diritto (triada CIA).
Autenticazione
Verifica dell'identità: qualcosa che sai (password), che hai (token), che sei (biometria). Il SO gestisce l'autenticazione per il login; le password sono memorizzate come hash (bcrypt, SHA-256 + salt) mai in chiaro.
Access Control
ACL (Access Control List): per ogni risorsa, lista di (soggetto, permessi). Capability list: per ogni processo, lista di (risorsa, permessi). I permessi Unix (rwx per owner/group/other) sono una ACL semplificata.
Buffer Overflow
Scrivere oltre i limiti di un buffer sovrascrive dati adiacenti (return address, variabili). Permite l'esecuzione di codice arbitrario. Contromisure: stack canary, ASLR (Address Space Layout Randomization), NX bit (no-execute), safe languages.
Malware & Virus
Tipi: Virus (si replica allegandosi a programmi), Worm (si propaga autonomamente in rete), Trojan (si maschera da software legittimo), Ransomware (cifra i file e chiede riscatto), Rootkit (nasconde la propria presenza nel SO).
Protezione del kernel: rings & sandbox
L'hardware supporta livelli di privilegio (ring 0 = kernel, ring 3 = user). Il codice user non può eseguire istruzioni privilegiate. La sandbox isola processi tramite namespaces, seccomp, cgroups (base dei container Docker).
