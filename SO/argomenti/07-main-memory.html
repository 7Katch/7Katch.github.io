Spazio di Indirizzamento fondamento

Lo spazio di indirizzamento è l'insieme di tutti gli indirizzi di memoria che un processo può generare. Su un sistema a 32 bit è 2³² = 4 GB; su un sistema a 64 bit è teoricamente 2⁶⁴, ma in pratica i SO usano 48 bit (256 TB) per indirizzamento virtuale. Lo spazio di indirizzamento di un processo è privato: ogni processo vede solo il proprio, isolato dagli altri.

2³²indirizzi a 32 bit = 4 GB
2⁴⁸indirizzi virtuali a 48 bit (usati in pratica)
4 KBdimensione tipica di una pagina

Lo spazio è diviso in due metà: la user space (dove risiedono codice, dati, stack e heap del processo) e la kernel space (mappata nello stesso spazio virtuale ma protetta — non accessibile dal codice utente). Su Linux a 64 bit la divisione è a metà dello spazio a 48 bit: 0x0000'0000'0000'0000 – 0x0000'7FFF'FFFF'FFFF per user, 0xFFFF'8000'0000'0000 – 0xFFFF'FFFF'FFFF'FFFF per kernel.

Layout tipico dello spazio di indirizzamento virtuale di un processo Linux a 64 bit.

Memoria Fisica, Logica e Virtuale concetti chiave

Memoria Fisica
La RAM vera, i chip fisici installati nel sistema.
Indirizzi fisici: prodotti dalla MMU dopo traduzione.
Limitata dalla RAM installata (es. 16 GB).
Condivisa tra tutti i processi e il kernel.
Memoria Logica
Lo spazio di indirizzi visto dal processo in esecuzione.
Indirizzi logici: generati dalla CPU durante l'esecuzione.
Ogni processo ha il proprio spazio logico separato.
La MMU traduce logico → fisico a runtime.
Memoria Virtuale
Astrazione che estende la memoria logica oltre la RAM fisica.
Usa il disco come estensione (swap/paging area).
Permette processi più grandi della RAM disponibile.
Gestita tramite demand paging e page replacement.

La MMU (Memory Management Unit) è l'unità hardware che traduce ogni indirizzo logico in fisico in modo trasparente. La traduzione avviene ad ogni accesso alla memoria:

MMU address translation
// CPU esegue: MOV rax, [0x40001234] ← indirizzo LOGICO
──────────────────────────────────────────────
indirizzo logico  0x40001234
pagina (bits 47..12)   0x40001  →  page table lookup
offset (bits 11..0)   0x234
──────────────────────────────────────────────
frame fisico trovato  0xB3
indirizzo fisico      0xB3234 = (0xB3 << 12) | 0x234
──────────────────────────────────────────────
✓ dato letto dalla RAM fisica all'indirizzo 0xB3234
💡 Il processo non sa mai dove i suoi dati risiedono fisicamente in RAM — conosce solo gli indirizzi logici. Questo è il principio fondamentale dell'astrazione della memoria.

Frammentazione Interna paginazione

La frammentazione interna si verifica quando la memoria assegnata a un processo è maggiore di quella effettivamente richiesta. Lo spazio in eccesso — pur all'interno del blocco allocato — non può essere usato da nessun altro processo, andando sprecato.

Con la paginazione a frame di dimensione fissa (es. 4 KB), se un processo richiede 13 KB, riceve 4 frame (16 KB). I 3 KB finali del quarto frame sono frammentazione interna: appartengono al processo, ma sono inutilizzati.

Spreco medio framm. interna media = page_size / 2 Con page_size = 4 KB → spreco medio ≈ 2 KB per processo

Ridurre la page size diminuisce la frammentazione interna, ma aumenta la dimensione delle page table e riduce l'efficienza del TLB. I SO moderni usano sia 4 KB che huge pages (2 MB o 1 GB) per workload specifici (database, macchine virtuali).

Frammentazione Esterna allocazione contigua

La frammentazione esterna è il problema opposto: la memoria libera totale è sufficiente per soddisfare una richiesta, ma non è fisicamente contigua. Esistono molti piccoli buchi sparsi tra i processi allocati, nessuno dei quali abbastanza grande da solo.

È il problema principale dell'allocazione contigua (dove ogni processo deve occupare un blocco unico di RAM consecutiva). Dopo molti cicli di allocazione/deallocazione, la RAM si riduce in frammenti inutilizzabili.

La soluzione tradizionale è la compattazione: spostare tutti i processi verso un'estremità della RAM per unire tutti i buchi in uno solo. È efficace ma costosa — richiede di copiare potenzialmente gigabyte di dati e di aggiornare tutti i puntatori. La soluzione moderna è la paginazione, che elimina il requisito di contiguità fisica.

⚠️ La regola del 50%: con strategia First-fit, statisticamente circa un terzo dei blocchi va sprecato per frammentazione esterna. Per ogni 2 blocchi allocati, ce n'è circa 1 di frammento libero inutilizzabile.

Paginazione & Page Table soluzione chiave

La paginazione divide sia la memoria fisica che lo spazio logico in blocchi di dimensione fissa. La RAM fisica è suddivisa in frame; lo spazio logico di ogni processo in pagine della stessa dimensione. Una page table (una per processo) mappa ogni numero di pagina al frame fisico corrispondente. La paginazione elimina completamente la frammentazione esterna — le pagine possono stare in frame qualsiasi, sparsi per la RAM.

Struttura dell'indirizzo logico

Un indirizzo logico a 32 bit con page size 4 KB (2¹²) si divide:

bit 31..12 Page Number (p)
bit 11..0 Offset (d) — 12 bit = 4096 byte

La MMU legge la page table[p] → ottiene il frame fisico f.
Indirizzo fisico = f × 4096 + d

La page table è una struttura dati che il kernel mantiene in RAM per ogni processo. Con pagine da 4 KB su 32 bit, una page table ha 2²⁰ = 1M entry. Con entry da 4 byte ciascuna, la page table occupa 4 MB per processo — troppo. Per questo i SO moderni usano page table multilivello (2 o 4 livelli su x86-64), allocando solo le parti della page table effettivamente usate.

⚡ TLB — Translation Lookaside Buffer
1. La CPU genera un indirizzo logico con numero di pagina p.
2. La MMU cerca p nel TLB (cache hardware, ~1 ciclo di clock). Se trovato → TLB hit: frame fisico disponibile immediatamente.
3. Se non trovato → TLB miss: accesso alla page table in RAM (~100 cicli). Il risultato viene inserito nel TLB.
4. Con hit rate tipico >99%, l'overhead medio è <1%.
5. Al context switch: TLB flush (o uso di ASID su ARM per evitare il flush).

Demand Paging & Page Fault memoria virtuale

Il demand paging è la tecnica che permette di caricare in RAM solo le pagine di un processo che sono effettivamente richieste, invece di caricare l'intero programma all'avvio. Le pagine non ancora caricate risiedono su disco (nella swap area o nel file eseguibile) e vengono portate in RAM solo al primo accesso.

✅ Il demand paging permette di eseguire processi più grandi della RAM disponibile, e di avviare i programmi più velocemente (solo le pagine necessarie vengono caricate subito).

Ogni entry della page table ha un valid/invalid bit. Se il processo accede a una pagina con bit = invalid, la MMU genera un'eccezione hardware: il page fault.

🔄 Sequenza gestione Page Fault
1. Il processo accede a un indirizzo logico → la pagina ha valid bit = 0.
2. La MMU genera un'eccezione di page fault. Il controllo passa al kernel.
3. Il kernel verifica: è un accesso valido (la pagina appartiene al processo)? Se no → segmentation fault, il processo viene terminato.
4. Il kernel cerca un frame libero in RAM. Se non c'è → page replacement (scegli una pagina vittima da rimuovere).
5. Il kernel carica la pagina mancante da disco nel frame trovato.
6. Aggiorna la page table: imposta il frame number e valid bit = 1.
7. Riprende l'esecuzione del processo dall'istruzione che aveva causato il fault.
kernel: page fault handler
// processo 1234 accede a 0x00C04000
page_fault(pid=1234, addr=0x00C04000)
  // check VMA: indirizzo appartiene al processo?
  vma lookup → OK (heap region)
  // cerca frame libero
  free_frame → frame #42
  // carica pagina da swap
  swap_in → disk block 0x1F2 → frame #42 [~5ms]
  // aggiorna page table
  pte[0xC04] = { frame:42, valid:1, dirty:0 }
  // riprende il processo
resume pid=1234 @ 0x00C04000
⚠️ Il page fault richiede un accesso al disco (~5-10 ms), mentre un normale accesso RAM è ~100 ns — un fattore 50.000x più lento. Minimizzare i page fault è critico per le prestazioni.

Page Replacement quando la RAM è piena

Quando la RAM è piena e occorre caricare una nuova pagina, il kernel deve scegliere una pagina vittima da rimuovere. Se la pagina vittima è stata modificata (il suo dirty bit è 1), deve essere scritta su disco prima di essere sovrascritta. Se non è stata modificata, può essere scartata direttamente.

Obiettivo: minimizzare page fault
L'algoritmo ottimale (OPT) rimpiazza la pagina che non verrà usata per il tempo più lungo. Non implementabile in tempo reale (richiede conoscenza del futuro).
Usato come benchmark teorico per confrontare algoritmi reali.
Un buon algoritmo di replacement si avvicina il più possibile a OPT.
Dirty bit & costo
Se dirty=0: la pagina è identica alla copia su disco → basta marcarla libera (0 I/O).
Se dirty=1: la pagina è stata modificata → va scritta su disco prima di liberarla (~5 ms).
Un buon algoritmo preferisce vittime con dirty=0 per ridurre gli I/O.

Esistono due tipi di page replacement: locale (la pagina vittima appartiene allo stesso processo che ha causato il fault) e globale (la vittima può appartenere a qualsiasi processo). Il replacement globale è usato da Linux e Windows perché permette di adattare dinamicamente la quantità di frame per processo.

Algoritmi: LRU & Seconda Chance implementazione

L'algoritmo LRU (Least Recently Used) rimpiazza la pagina che non è stata usata da più tempo. Si basa sul principio di locality of reference: le pagine usate di recente hanno più probabilità di essere usate ancora a breve. LRU approssima bene l'algoritmo OPT.

Esempio: 3 frame disponibili, sequenza di accessi: 7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3

Accesso Frame 0 Frame 1 Frame 2 Evento Vittima
7 7 — — FAULT —
0 7 0 — FAULT —
1 7 0 1 FAULT —
2 2 0 1 FAULT evict 7
0 2 0 1 HIT —
3 2 0 3 FAULT evict 1
0 2 0 3 HIT —
4 4 0 3 FAULT evict 2
2 4 2 3 FAULT evict 0
3 4 2 3 HIT —
0 0 2 3 FAULT evict 4
3 0 2 3 HIT —

Totale: 8 page fault su 12 accessi. HIT rate = 33%.

LRU puro richiede di marcare ogni accesso a ogni pagina con un timestamp — impossibile implementarlo efficientemente in hardware. Si usano due approssimazioni pratiche:

🔄 Algoritmo Seconda Chance (Clock Algorithm)
1. I frame sono organizzati in un buffer circolare (come un orologio). Ogni frame ha un reference bit (R), settato a 1 dall'hardware ogni volta che la pagina viene acceduta.
2. Quando serve una vittima, la lancetta dell'orologio avanza.
3. Se la pagina puntata ha R=1: seconda chance — azzera R a 0, avanza la lancetta senza rimpiazzare. La pagina ottiene un'altra opportunità.
4. Se la pagina puntata ha R=0: è la vittima — viene rimpiazzata.
5. Nel caso peggiore (tutti R=1), la lancetta fa un giro completo e rimpiazza la prima pagina (ora R=0).

La lancetta punta a p4 (R=0): è la vittima. Le pagine con R=1 ricevono seconda chance (R→0) e la lancetta avanza.

Frequenza dei Page Fault controllo del thrashing

La frequenza dei page fault (PFF) misura quanti page fault per unità di tempo genera un processo. È l'indicatore principale per decidere se un processo ha troppa poca o troppa RAM.

Il kernel usa la PFF per il frame allocation control:

PFF troppo alta → thrashing
Il processo genera page fault così frequentemente che passa più tempo ad aspettare I/O che ad eseguire.
Il kernel alloca più frame al processo, o sospende altri processi per liberare RAM.
Se tutto il sistema è in thrashing → swap storm: la CPU è quasi inattiva mentre il disco è al 100%.
PFF troppo bassa → spreco
Il processo ha più frame di quanti ne usa il suo working set.
Il kernel può riallocare frame a processi che ne hanno più bisogno.
Indica potenziale per caricare più processi in RAM simultaneamente (aumento del grado di multiprogrammazione).
Tempo effettivo di accesso EAT = (1−p)·t_mem + p·t_fault p=prob. page fault, t_mem≈100ns, t_fault≈10ms → anche 0.1% di fault → EAT 100× peggiore

Working Set Model locality

Il working set model si basa sul principio di locality of reference: durante qualsiasi finestra temporale, un processo usa solo un piccolo sottoinsieme delle sue pagine. Il working set W(t, Δ) è l'insieme delle pagine usate nelle ultime Δ unità di tempo (la finestra temporale).

La dimensione del working set |W(t, Δ)| indica quanti frame il processo ha bisogno in questo momento per girare senza thrashing. Se la somma dei working set di tutti i processi supera il numero di frame disponibili → thrashing inevitabile.

Sequenza di riferimenti a pagine: 1, 2, 1, 5, 7, 7, 1, 3, 4, 3, 4, 4, 3, 4, 5, 4, 1, 6, 3, 7
Finestra temporale Δ = 5 accessi. La finestra scorre sull'accesso corrente (cursore giallo):

📐 Working Set → Frame Allocation
1. Il kernel stima periodicamente |W(t, Δ)| per ogni processo tramite i reference bit (approssimazione con interrupt periodici).
2. Calcola D = Σ|Wi| (somma dei working set di tutti i processi).
3. Se D > numero frame totali → thrashing. Sospendi un processo per liberare i suoi frame.
4. Se D << frame totali → carica un nuovo processo per aumentare l'utilizzo della CPU.
5. Linux implementa questo con la swap tendency e il kswapd daemon che agisce in base alla memoria libera disponibile.
🎯 Il working set cambia nel tempo man mano che il programma entra in fasi diverse (es. inizializzazione → loop principale → I/O → cleanup). Il kernel deve tracciarlo dinamicamente e riallocare i frame di conseguenza.
