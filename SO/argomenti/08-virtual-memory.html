Demand Paging fondamento

La memoria virtuale è l'illusione di uno spazio di indirizzi molto più grande della RAM fisica disponibile. Il principio: non tutte le pagine di un processo devono stare in RAM contemporaneamente. Il demand paging carica le pagine in memoria solo quando vengono effettivamente accedute (lazy loading).

Ogni entry della page table ha un valid bit: se è 1, la pagina è in RAM e l'indirizzo del frame è valido. Se è 0, la pagina non è in RAM (è su disco o non è stata ancora allocata). Il vantaggio è significativo: un programma può avere uno spazio virtuale di 48 bit (256 TB su x86-64) anche con soli 16 GB di RAM fisica. Più processi possono girare contemporaneamente perché ognuno usa solo la RAM strettamente necessaria.

Page Fault

Quando il processore accede a una pagina con valid bit = 0, genera una page fault: una trap che trasferisce il controllo al gestore di page fault del kernel. La sequenza di gestione è: il SO verifica che l'accesso sia legittimo (la pagina appartiene allo spazio virtuale del processo — altrimenti è un segfault). Il SO trova un frame libero in RAM. Il SO carica la pagina dal disco (area di swap o file ELF). Il SO aggiorna la page table (valid bit = 1, numero frame). Il SO riprende l'istruzione che aveva causato il fault.

Il costo è enorme: un accesso in RAM richiede ~100 ns, un page fault con accesso a SSD richiede ~100 µs, su HDD ~10 ms. Per questo il tasso di page fault deve essere bassissimo: anche un solo page fault ogni 1000 accessi fa crollare le prestazioni del 99%.

Sostituzione delle Pagine

Quando tutti i frame sono occupati e serve caricarne una nuova, il SO deve scegliere una pagina vittima da espellere su disco. Se la pagina è stata modificata (dirty bit = 1), va scritta su disco prima di liberare il frame. Se è clean, il frame si libera direttamente.

Gli algoritmi di sostituzione si valutano per tasso di page fault (minore = migliore). FIFO: si espelle la pagina in RAM da più tempo. Semplice ma pessimo: espelle spesso pagine molto usate. Soffre dell'anomalia di Belady (aggiungere frame può aumentare i page fault). OPT (Ottimale): si espelle la pagina che non verrà usata per più tempo in futuro. Impossibile da implementare (richiede di vedere il futuro), ma serve come benchmark teorico. LRU (Least Recently Used): si espelle la pagina non usata da più tempo. Ottima approssimazione dell'OPT, ma costosa da implementare esattamente (richiede hardware o lista ordinata aggiornata a ogni accesso). Clock (Second-Chance): approssimazione LRU pratica usata da Linux. Le pagine sono in un buffer circolare con un bit di riferimento. L'algoritmo gira come un orologio: se trova una pagina con bit = 0 la espelle; se trova bit = 1 lo azzera e passa avanti.

Allocazione dei Frame

Quanti frame assegnare a ogni processo? Con troppo pochi frame il processo genera page fault continui. Con troppi, si toglie RAM agli altri processi. Le strategie principali sono tre.

Equal allocation: dividere i frame equamente tra tutti i processi. Semplice ma ingiusto: un processo piccolo non ha bisogno degli stessi frame di uno grande. Proportional allocation: ogni processo riceve un numero di frame proporzionale alla sua dimensione. Priority allocation: i processi con priorità più alta ricevono più frame, a scapito di quelli con priorità bassa. In tutti i casi esiste un numero minimo di frame (dipende dall'architettura ISA): deve essere sufficiente per completare almeno un'istruzione anche nel caso peggiore di page fault.

Thrashing

Il thrashing è una condizione patologica in cui il sistema passa più tempo a gestire page fault che a eseguire codice utile. Si verifica quando un processo ha meno frame di quanti ne servono per il suo working set attuale: espelle una pagina che sarà richiesta quasi subito, generando un page fault quasi immediato.

Se il sistema ha molti processi in thrashing, la CPU utilization crolla nonostante ci siano processi "in esecuzione". Lo scheduler vede processi ready e ne aggiunge altri, peggiorando la situazione. Il SO deve rilevare il thrashing (monitorando il tasso di page fault o la CPU utilization) e reagire riducendo il grado di multiprogramming: sospendere alcuni processi su disco (swap out) per liberare frame per gli altri.

Working Set Model

Il working set di un processo è l'insieme delle pagine accedute nelle ultime Δ referenze di memoria (Δ è la finestra temporale, un parametro configurabile). Rappresenta la località di riferimento attuale del processo: le pagine che usa intensivamente in questo momento.

Il principio del working set: se un processo riceve abbastanza frame da contenere il suo working set, il tasso di page fault sarà basso. Se riceve meno frame del working set thrashing. Il SO può monitorare la somma dei working set di tutti i processi: se supera i frame totali disponibili, il sistema è in pericolo di thrashing e va ridotto il grado di multiprogramming.

Memory-Mapped Files

Invece di leggere un file con read() in un buffer separato, mmap() mappa il file direttamente nello spazio di indirizzi virtuale del processo. Accedere ai dati del file è come accedere a un array in memoria: un semplice pointer dereference. Il kernel gestisce il caricamento delle pagine del file tramite demand paging.

I vantaggi sono notevoli: eliminazione delle copie kerneluser (zero-copy in certi scenari), condivisione efficiente tra processi dello stesso file (la stessa pagina fisica), e semplicità di codice. È la base di molte librerie condivise (mappate in read-only da tutti i processi che le usano) e di database che gestiscono la propria cache (come PostgreSQL con shared_buffers).
