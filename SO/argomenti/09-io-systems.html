    Categorie di Dispositivi I/O

    L'incredibile varietà di periferiche hardware obbliga il SO a usare i Device Drivers per incapsulare i dettagli tecnici in interfacce uniformi chiamate System Calls. I dispositivi si suddividono principalmente in:

    • Block Devices: Es. Hard Disk. I comandi base sono read, write, seek su blocchi di byte definiti. Usano ampiamente il DMA.
    • Character (Stream) Devices: Es. Tastiere, Mouse, Porte Seriali. I comandi base sono get(), put(). Leggono/scrivono un carattere alla volta, in un flusso.
    • Network Devices: Es. Schede di rete. Hanno un'interfaccia a sé stante (i Socket), che separa il protocollo di rete dall'operazione in sé.
    • Clocks & Timers: Forniscono l'orario corrente, timer per interrompere ciclicamente la CPU, corretti spesso dal protocollo di rete NTP (Network Time Protocol).

    In UNIX/Linux, tutti questi dispositivi sono presentati e astratti all'utente come file speciali dentro la directory /dev.

    Componenti I/O Hardware

    Un dispositivo I/O si interfaccia col computer tramite tre componenti hardware principali:

    A Porta: il punto di connessione fisica per il dispositivo (es. USB, porta seriale).
    B Bus: il gruppo di fili condivisi (es. PCI/PCIe) che collega più dispositivi instradando i dati alla CPU.
    C Controller: L'elettronica (chipset dedicato) che gestisce la porta e il bus.

    I Registri del Controller

    Il Device Driver interagisce con il Controller scrivendo/leggendo bit nei suoi Registri Interni, mappati spesso nello spazio di indirizzamento della memoria (Memory-mapped I/O). Questi registri sono tipicamente 4:

    1. Data-in Register: Dati letti dall'host (il computer).
    2. Data-out Register: Dati scritti dall'host.
    3. Status Register: Bit che indicano lo stato attuale (es. dispositivo occupato/libero, errore occorso).
    4. Control Register: L'host vi scrive comandi per modificare la modalità operativa.

    Comunicazione: Polling vs Interrupts

    Come fa il SO a capire che un dispositivo ha finito un'operazione?

    Polling (Busy-Wait)

    Il protocollo più elementare. L'host (CPU) interroga continuamente il bit di stato (busy bit) del dispositivo in un ciclo chiuso (while loop).

    1. Host legge il busy_bit continuamente finché è 0.
    2. L'host imposta il comando (read/write) nel command_register.
    3. L'host imposta il bit di command-ready.
    4. Il controller alza il busy_bit a 1 e inizia a lavorare.
    5. Finito il lavoro, il controller riabbassa il busy_bit a 0.

    Contro: Spreca enormi quantità di cicli CPU (inefficiente) se il dispositivo è molto più lento della CPU, perché costringe il processore ad aspettare girando a vuoto.

    Interrupts (Interrupt Request Line)

    Per risolvere l'inefficienza del Polling, la CPU delega il lavoro e torna a fare altro (controllando un'apposita linea elettrica hardware al termine di ogni singola istruzione). Quando il dispositivo I/O finisce, manda un segnale elettrico. La CPU si ferma, fa un context switch e chiama l'Interrupt Handler corretto basandosi sulla sua posizione nella Interrupt Vector Table.

    Esistono Maskable Interrupts (disattivabili dalla CPU prima di eseguire istruzioni critiche) e Non-maskable (non ignorabili, es. memorie guaste e system faults vitali).

    Direct Memory Access (DMA)

    Le periferiche veloci o i dischi che muovono blocchi immensi di dati (Block Devices) intaserebbero rapidamente il sistema se la CPU dovesse farsi carico di ogni singolo byte (Programmed I/O).

    La soluzione è il DMA Controller (DMAC), un chip dedicato che bypassa la CPU.

    1. Il SO prepara in memoria (RAM) un "DMA command block", contenente:
      Indirizzo sorgente, Indirizzo destinazione, Read/Write, e il Numero di Byte (count).
    2. Il SO dice al controller I/O di avviare il DMA fornendogli il puntatore a questo blocco.
    3. Il DMAC ruba il bus (Bus Stealing) alla CPU e trasferisce autonomamente l'intero mucchio di byte dal dispositivo hardware direttamente alla RAM (o viceversa).
    4. Terminato il trasferimento di tutto il blocco, il DMA invia un singolo Interrupt alla CPU per notificare il successo.
    Differenza Critica: Con l'I/O Programmato c'è un interrupt per ogni singolo byte letto. Col DMA, c'è un solo interrupt per l'intero blocco di dati!

    Modelli di Comunicazione

    A livello di codice utente/applicazione, esistono 3 grandi modelli di I/O gestiti dal Kernel:

    Blocking I/O (Sincrono)
    Il processo viene sospeso/bloccato dall'OS finché l'intera operazione I/O hardware non è giunta al termine.
    Molto facile e logico da programmare, ma impedisce al programma di fare altre computazioni nell'attesa.
    Non-blocking & Async
    Non-blocking: La chiamata di sistema ritorna ISTANTANEAMENTE con i byte effettivamente pronti (anche 0), ad es. mouse o tastiera (se non si preme niente, l'app non si congela).
    Asynchronous: Il processo esegue l'I/O (su disco o rete) che lavorerà in background. Il processo continua l'esecuzione. L'OS manderà un segnale al processo, *nel futuro*, quando l'operazione in backgound sarà completata con successo.

    Error Handling e I/O Protection sicurezza

    Essendo i dispositivi di I/O estremamente eterogenei e inclini al fallimento fisico, il SO è programmato per gestire errori transitori (es. lettura temporaneamente fallita da un disco di rete) ritentando automaticamente, oppure riportando un codice d'errore all'applicazione e salvando il report in un system log.

    I/O = Istruzioni Privilegiate: Poiché un programma utente potrebbe accidentalmente o volutamente distruggere dati usando istruzioni I/O malevole, TUTTE le istruzioni I/O hardware sono definite come privilegiate. Un processo utente NON PUÒ interagire mai direttamente con i dispositivi, ma deve sempre passare attraverso l'OS invocando una System Call (che effettua la transizione sicura da User Mode a Kernel Mode).
