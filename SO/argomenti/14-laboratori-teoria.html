    Laboratori (Teoria)

    Una sintesi dei concetti teorici fondamentali estratti dalle sessioni di laboratorio (C su Linux e RIOT OS).

    7Katch.github.io/ SO/ 14 · Laboratori

    Kernel Modules e Privilegi

    I programmi standard (eseguibili scritti in C) girano in User Mode. Tuttavia, in Linux è possibile scrivere programmi che si interfacciano direttamente con il Kernel tramite i Kernel Modules, eseguiti in Kernel Mode. Questo implica il massimo livello di privilegio (e il rischio di causare un crash dell'intero sistema, Kernel Panic, se il codice contiene bug).

    • Entry & Exit Points: Un modulo kernel non ha il classico main(). Utilizza funzioni specifiche (es. simple_init() e simple_exit()) che vengono invocate quando il modulo viene caricato in memoria o scaricato. Tali funzioni vengono registrate usando le macro module_init() e module_exit().
    • Stampa di sistema: Poiché la libreria standard C (come stdio.h e la funzione printf) è spazio utente, in Kernel mode si utilizza printk() (Kernel Print). L'output non finisce sul terminale corrente, ma nel Kernel Log Buffer, che può essere letto con il comando dmesg.

    Processi, Thread e Contention Scope

    Processi Orfani (Orphan Processes)

    Quando un processo crea un figlio (tramite la syscall fork()), se il processo Padre termina l'esecuzione prima del processo Figlio, quest'ultimo diventa un processo Orfano. In Linux, i processi orfani vengono automaticamente adottati da systemd (il primissimo processo inizializzato dal sistema operativo, con PID = 1).

    Libreria Pthread e Tipi di Parallelismo

    La creazione di thread in Linux usa lo standard POSIX Threads (pthreads). Condividendo i dati, i thread pongono problemi di Race Condition, ma aprono le porte al calcolo parallelo:

    • Data Parallelism: Molteplici thread eseguono la stessa identica operazione ma su partizioni di dati diversi (es. un array diviso in chunk da elaborare simultaneamente).
    • Task Parallelism: Molteplici thread eseguono operazioni distinte e uniche.

    Contention Scope (SCS vs PCS)

    Lo scheduling dei thread può avvenire a due livelli (Scope):

    System-Contention Scope (SCS): I thread competono per l'utilizzo della CPU a livello dell'intero sistema. Il Kernel schedula direttamente i kernel thread sulla CPU fisica disponibile. È il modello standard (e unico permesso) in Linux e macOS.

    Process-Contention Scope (PCS): (Modelli many-to-many o many-to-one). I thread utente vengono schedulati dalla thread library locale del processo per girare su un Lightweight Process (LWP). La competizione avviene solo tra i thread di quello specifico processo.

    Strumenti di Sincronizzazione in C

    Oltre alle basi teoriche di Peterson's Solution e dell'algoritmo del banchiere per evitare i Deadlock (gestiti analiticamente), il codice POSIX offre costrutti chiave:

    • POSIX Mutex: pthread_mutex_t fornisce la mutua esclusione. Le funzioni pthread_mutex_lock() e unlock() proteggono le sezioni critiche. Rispetto a uno spinlock (che spreca cicli di CPU in busy-waiting), un thread bloccato su un mutex bloccato da un altro thread viene sospeso finché non si libera.
    • Named vs Unnamed Semaphores: I semafori POSIX possono essere:
      • Unnamed (Memory-based): Nessun nome, allocati direttamente nella memoria condivisa tra i thread (es. variabile globale) o processi (es. shared memory object POSIX).
      • Named Semaphores: Hanno un identificatore a stringa (es. /miosenaforo). Questo permette a processi indipendenti di operare sullo stesso semaforo chiamando sem_open(). Bisogna ricordarsi di usare sem_close() e soprattutto sem_unlink() per distruggerli a livello di sistema, altrimenti (se il processo crascia) manterranno valori corrotti persistenti nel file system (in /dev/shm).

    Accesso in Memoria, MMIO e Link

    Struttura dei Programmi e Performance Memory-based

    Il modo in cui i dati vengono letti incide drasticamente sul numero di Page Fault e di miss nella cache o nella TLB. In C gli array sono allocati Row-major order (per riga). Se un programma legge i dati scorrendoli per colonna (Col-major), salterà in continuazione da una pagina di memoria a un'altra, annullando l'efficacia del fetch anticipato e aumentando enormemente le TLB/Cache miss e i page faults.

    Memory-Mapped I/O (mmap)

    La system call mmap() in Linux mappa un file direttamente nello spazio di indirizzamento (memoria virtuale) del processo. Questo permette al codice di trattare il file come un array in memoria:

    • Lazy Loading: Le pagine del file vengono caricate in memoria fisica (Demand Paging) solo quando il codice cerca effettivamente di accedervi.
    • No Syscall Overhead: Si elimina il costo delle system calls ripetitive (read() / write()) e il passaggio continuo tra kernel space e user space (eliminando l'overhead di copia nei buffer intermedi).

    File System (ext4 e Link)

    A differenza di vecchi File System ext2/ext3 che usavano puntatori indiretti combinati, l'attuale ext4 usa un Extent-based system: range di blocchi fisici contigui (fino a 128 MB di spazio per singolo extent), riducendo la complessità per file giganti.

    Fast Symlinks: Riguarda la distinzione tra Hard Link (puntatori diretti allo stesso Inode di un file su disco) e Symbolic Link (puntatori che indicano un percorso). In ext4, se il percorso del symbolic link è molto corto (sotto i 60 caratteri), viene salvato direttamente dentro il blocco dell'Inode del symlink stesso, evitando l'allocazione di un blocco dati (azzerando così la frammentazione e i tempi di lettura su disco). Questo prende il nome di Fast Symlink.

    RIOT: Real-Time OS per IoT

    I laboratori affrontano la programmazione di dispositivi Constrained (come i microcontrollori 8-bit Arduino Mega 2560) utilizzando RIOT OS. È un sistema basato su microkernel progettato esplicitamente per dispositivi di rete a bassa potenza.

    • Shell e Moduli: RIOT permette di compilare e agganciare moduli specifici a un'applicazione (es. USEMODULE += shell) per interagire con l'hardware e monitorare i thread in tempo reale con comandi come ps.
    • Gestione Thread: Per tenere un thread in background e farlo sopravvivere al thread principale che lancia la shell, occorre che possieda una priorità più bassa (valore numerico più alto) e che esegua un loop infinito, altrimenti terminerebbe immediatamente (e non apparirebbe in ps).
    • Timers Hardware (ztimer): Utilizza moduli aggiuntivi per temporizzare le operazioni (ztimer_sleep) al fine di sincronizzare le operazioni cicliche (es. lettura hardware) millisecondo per millisecondo, simulando esecuzioni concorrenti protette dai Mutex nativi del sistema real-time.
