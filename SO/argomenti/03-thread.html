Benefici del Multithreading fondamento

Un thread è un'unità di esecuzione all'interno di un processo. Tutti i thread di un processo condividono codice, dati globali e file aperti, ma ognuno ha il proprio stack, il proprio program counter e i propri registri. Questo li rende molto più leggeri dei processi.

I vantaggi del multithreading sono quattro: Responsività — un'applicazione può continuare a rispondere all'utente anche mentre un thread esegue un'operazione lunga (es. download). Condivisione delle risorse — i thread comunicano direttamente tramite la memoria del processo, senza IPC. Economia — creare un thread richiede molto meno overhead di una fork(). Scalabilità — su sistemi multicore i thread vengono distribuiti su più processori, ottenendo parallelismo reale.

Concorrenza vs Parallelismo

Concorrenza significa che più task sono in progresso nello stesso periodo di tempo, ma non necessariamente nello stesso istante. Anche su un singolo core, il SO può far avanzare più thread con time-slicing (interleaving), creando l'illusione della simultaneità.

Parallelismo significa che due o più task vengono eseguiti letteralmente nello stesso istante su core (o CPU) diversi. È possibile solo con hardware multicore. Il parallelismo implica concorrenza, ma non viceversa.

Per un programmatore, scrivere codice corretto in concorrenza è la sfida principale: race condition e deadlock possono presentarsi sia in modalità concorrente che parallela, ma diventano molto più difficili da riprodurre e debuggare con il vero parallelismo.

Parallelismo dei Dati e dei Task

Il parallelismo dei dati applica la stessa operazione su sottoinsiemi diversi di un grande insieme di dati, distribuiti su più core. Esempio: sommare un array di un milione di elementi dividendolo in 8 parti e assegnando ogni parte a un core separato.

Il parallelismo dei task distribuisce invece operazioni diverse su core diversi, spesso su dati diversi. Esempio: un webserver che gestisce richieste HTTP diverse su thread diversi, ognuno eseguendo la stessa sequenza di operazioni ma su una connessione distinta.

Nella pratica le applicazioni moderne combinano entrambi. Framework come OpenMP facilitano il parallelismo dei dati; librerie come pthreads vengono usate per il parallelismo dei task.

Legge di Amdahl

La Legge di Amdahl quantifica il limite teorico dello speedup ottenibile parallelizzando un programma con N core. La formula è: Speedup = 1 / (S + (1-S)/N), dove S è la frazione di codice che rimane seriale (non parallelizzabile) e N è il numero di core.

Il risultato è sorprendente: se solo il 5% del codice è seriale (S=0.05), anche con infiniti core lo speedup massimo è solo 20×. La legge mostra perché ridurre la parte seriale è fondamentale: portare S da 10% a 1% può triplicare il guadagno ottenibile dall'hardware parallelo.

Nella pratica, lock e sezioni critiche costituiscono la parte seriale nascosta di molti programmi. Un'analisi con strumenti come perf aiuta a identificarle.

User Thread e Kernel Thread

I user thread (thread a livello utente) sono gestiti da una libreria in user space (es. GNU Pthreads, Java thread). Il kernel non li conosce: vede solo il processo contenitore. Sono veloci da creare e da commutare (nessun cambio di modalità kernel), ma se un user thread si blocca in una syscall, blocca tutti gli altri thread del processo.

I kernel thread sono entità di scheduling del kernel. Il SO li conosce e può distribuirli su core diversi. Se un kernel thread si blocca in I/O, il kernel può schedulare un altro thread dello stesso processo su un altro core. Il loro costo di creazione e context switch è maggiore degli user thread.

Modelli di Multithreading

I modelli di multithreading descrivono come i thread utente vengono mappati sui kernel thread. La scelta del modello determina il grado di parallelismo reale e l'overhead di gestione.

Many-to-One

Nel modello Many-to-One, molti user thread sono mappati su un singolo kernel thread. La libreria thread gestisce lo scheduling tra gli user thread. Il vantaggio è la semplicità e l'efficienza della commutazione (tutto in user space). Lo svantaggio è pesante: se un thread chiama una syscall bloccante, tutti i thread del processo si bloccano, e non c'è mai vero parallelismo. Oggi questo modello è raro nei sistemi general-purpose.

One-to-One

Nel modello One-to-One, ogni user thread è mappato su un distinto kernel thread. Il kernel può schedulare thread diversi su core diversi in parallelo. Se un thread si blocca in I/O, gli altri continuano a girare. È il modello usato da Linux (tramite clone()) e da Windows. Lo svantaggio è che creare molti thread ha un costo maggiore, poiché ogni user thread richiede un kernel thread.

Many-to-Many

Nel modello Many-to-Many, N user thread vengono mappati su M kernel thread, con M ≤ N. Il programmatore può creare quanti user thread vuole; il kernel ne mantiene un numero adeguato. Quando un user thread si blocca, un altro può usare il kernel thread libero. È il modello più flessibile ma anche il più complesso da implementare. Alcune implementazioni lo combinano con il modello Two-level: alcuni thread critici sono mappati One-to-One, gli altri Many-to-Many.
