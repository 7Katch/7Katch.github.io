    Benefici del Multithreading fondamento

    definizione

    Un thread è un'unità di esecuzione all'interno di un processo. Tutti i thread di un processo condividono codice, dati globali e file aperti, ma ognuno ha il proprio stack, il proprio program counter e i propri registri. Questo li rende molto più leggeri dei processi.

    I vantaggi del multithreading sono quattro: Responsività — un'applicazione può continuare a rispondere all'utente anche mentre un thread esegue un'operazione lunga (es. download). Condivisione delle risorse — i thread comunicano direttamente tramite la memoria del processo, senza IPC. Economia — creare un thread richiede molto meno overhead di una fork(). Scalabilità — su sistemi multicore i thread vengono distribuiti su più processori, ottenendo parallelismo reale.

    Concorrenza vs Parallelismo

    Concorrenza significa che più task sono in progresso nello stesso periodo di tempo, ma non necessariamente nello stesso istante. Anche su un singolo core, il SO può far avanzare più thread con time-slicing (interleaving), creando l'illusione della simultaneità.

    Parallelismo significa che due o più task vengono eseguiti letteralmente nello stesso istante su core (o CPU) diversi. È possibile solo con hardware multicore. Il parallelismo implica concorrenza, ma non viceversa.

    Per un programmatore, scrivere codice corretto in concorrenza è la sfida principale: race condition e deadlock possono presentarsi sia in modalità concorrente che parallela, ma diventano molto più difficili da riprodurre e debuggare con il vero parallelismo.

    Parallelismo dei Dati e dei Task

    Il parallelismo dei dati applica la stessa operazione su sottoinsiemi diversi di un grande insieme di dati, distribuiti su più core. Esempio: sommare un array di un milione di elementi dividendolo in 8 parti e assegnando ogni parte a un core separato.

    Il parallelismo dei task distribuisce invece operazioni diverse su core diversi, spesso su dati diversi. Esempio: un webserver che gestisce richieste HTTP diverse su thread diversi, ognuno eseguendo la stessa sequenza di operazioni ma su una connessione distinta.

    Nella pratica le applicazioni moderne combinano entrambi. Framework come OpenMP facilitano il parallelismo dei dati; librerie come pthreads vengono usate per il parallelismo dei task.

    Legge di Amdahl

    La Legge di Amdahl quantifica il limite teorico dello speedup ottenibile parallelizzando un programma con N core. La formula è:

    Formula legge di Amdahl
    \begin{equation} Speedup = \frac{1}{S + \frac{1 - S}{N}} \end{equation}

    dove

    • S è la porzione di codice (in %) che rimane seriale (non parallelizzabile)
    • N è il numero di core.

    Il risultato è sorprendente: se solo il 5% del codice è seriale (S=0.05), anche con infiniti core lo speedup massimo è solo 20x. La legge mostra perché ridurre la parte seriale è fondamentale: portare S da 10% a 1% può triplicare il guadagno ottenibile dall'hardware parallelo.

    Nella pratica, lock() e sezioni critiche costituiscono la parte seriale nascosta di molti programmi. Un'analisi con strumenti come perf aiuta a identificarle.

    User Thread e Kernel Thread

    I user thread (thread a livello utente) sono gestiti da una libreria in user space (es. GNU Pthreads, Java thread). Il kernel non li conosce: vede solo il processo contenitore. Sono veloci da creare e da commutare (nessun cambio di modalità kernel), ma se un user thread si blocca in una syscall, blocca tutti gli altri thread del processo.



    I kernel thread sono entità di scheduling del kernel. Il SO li conosce e può distribuirli su core diversi. Se un kernel thread si blocca in I/O, il kernel può schedulare un altro thread dello stesso processo su un altro core. Il loro costo di creazione e context switch è maggiore degli user thread.

    Modelli di Multithreading

    I modelli di multithreading descrivono come i thread utente vengono mappati sui kernel thread. La scelta del modello determina il grado di parallelismo reale e l'overhead di gestione.

    Many-to-One

    Nel modello Many-to-One, molti user thread sono mappati su un singolo kernel thread. La libreria thread gestisce lo scheduling tra gli user thread. Il vantaggio è la semplicità e l'efficienza della commutazione (tutto in user space). Lo svantaggio è pesante: se un thread chiama una syscall bloccante, tutti i thread del processo si bloccano, e non c'è mai vero parallelismo. Oggi questo modello è raro nei sistemi general-purpose.

    One-to-One

    Nel modello One-to-One, ogni user thread è mappato su un distinto kernel thread. Il kernel può schedulare thread diversi su core diversi in parallelo. Se un thread si blocca in I/O, gli altri continuano a girare. È il modello usato da Linux (tramite clone()) e da Windows. Lo svantaggio è che creare molti thread ha un costo maggiore, poiché ogni user thread richiede un kernel thread.

    Many-to-Many

    Nel modello Many-to-Many, N user thread vengono mappati su M kernel thread, con M ≤ N. Il programmatore può creare quanti user thread vuole; il kernel ne mantiene un numero adeguato. Quando un user thread si blocca, un altro può usare il kernel thread libero. È il modello più flessibile ma anche il più complesso da implementare. Alcune implementazioni lo combinano con il modello Two-level: alcuni thread critici sono mappati One-to-One, gli altri Many-to-Many.

    Librerie Thread (Pthreads)

    I thread a livello utente sono gestiti tramite API fornite da specifiche librerie. La più nota in ambiente UNIX (Linux, macOS) è Pthreads (POSIX Threads, IEEE 1003.1c). È importante notare che Pthreads è solo una specifica, non un'implementazione (che è demandata al creatore della libreria o del SO). In ambiente Microsoft esiste invece la Windows Thread API.

    Nel mondo reale e nei laboratori, Pthreads si usa per creare programmi multithreading in C. Di seguito l'idioma tipico: 

    #include <pthread.h>
#include <stdio.h>

int sum = 0; // Dato condiviso tra i thread

void *runner(void *param) {
    /* Il thread esegue questo codice */
    sum += 1;
    pthread_exit(0);
}

int main(int argc, char *argv[]) {
    pthread_t tid; // Identificatore del thread
    pthread_attr_t attr; // Attributi del thread

    pthread_attr_init(&attr);
    // Creazione del thread, esecuzione della funzione runner
    pthread_create(&tid, &attr, runner, argv[1]);
    
    // Il processo aspetta la terminazione del thread
    pthread_join(tid, NULL);
    printf("Sum = %d\n", sum);
}

    Nel codice sopra, il problema critico è che sum è una risorsa condivisa. Senza adeguate protezioni, se più thread accedono e modificano contemporaneamente la variabile, si verifica una race condition.

    Implicit Threading

    Creare e gestire esplicitamente centinaia di thread in applicazioni complesse diventa impraticabile e propenso ad errori (bug, race conditions sfuggenti). La soluzione è l'Implicit Threading: il programmatore identifica i task parallelizzabili, ma lascia la creazione e la gestione dei thread al compilatore e alle librerie di runtime.

    Soluzioni Principali
    1. Fork-Join: il thread principale si biforca e poi attende.
    2. Thread-pools: riutilizzo di thread preesistenti.
    3. OpenMP: direttive del compilatore C/C++ (#pragma omp parallel) che parallelizzano automaticamente i blocchi di codice in base ai core disponibili.
    4. Apple GCD (Grand Central Dispatch): tecnologia per macOS/iOS che accoda task in dispatch queue gestite da un thread pool del sistema operativo.
    5. Intel TBB: template library per programmazione parallela in C++.

    Modello Fork-join

    Il thread principale (parent) "biforca" (fork) uno o più thread figli per eseguire task concorrenti, per poi chiamare una join() e attendere la loro terminazione prima di proseguire. È un modello sincrono ed è l'equivalente per i thread del fork-wait usato per i processi.

    Modello Thread pools

    Vengono creati un numero finito di threads in attesa a lavorare.

    I vantaggi sono i seguenti

    • Solitamente più efficiente assegnare ad un thread già esistente che a crearne uno nuovo
    • Permette di vincolare il numero di thread nell'applicazione (o nelle applicazioni) alla dimensione del pool.
    • Separare il compito (task) da eseguire dai meccanismi di creazione del task stesso facilita diverse strategie di esecuzione:
    • Ad esempio, le API di Windows supportano i pool di thread.

    Problemi con il Threading

    Lo sviluppo di applicazioni multithread comporta diverse sfide architetturali:

    1. Semantica di fork() e exec(): Se un thread chiama fork(), il nuovo processo duplica tutti i thread o solo quello chiamante? Spesso UNIX offre due versioni della system call per gestire entrambi gli scenari.
    2. Gestione dei segnali: In un processo a singolo thread i segnali (software interrupt) vengono gestiti in modo diretto. Con più thread, bisogna decidere a quale thread recapitare il segnale (es. a chi l'ha generato, a tutti, o a uno dedicato).
    3. Cancellazione Thread: (Target thread cancellation) Un thread può essere terminato asincronamente in modo immediato (pericoloso per i dati condivisi) o in modalità differita (Deferred), dove il thread controlla periodicamente un flag per capire se deve autoterminarsi in modo sicuro.
    4. Thread-local-storage (TLS): Memoria visibile a livello globale tra le funzioni dello stesso thread, ma invisibile e separata rispetto agli altri thread. Serve quando i thread non devono condividere specifiche variabili (es. seed di un generatore casuale).
    5. Attivazioni scheduler: Modelli come il Many-to-Many richiedono una comunicazione tra user thread e kernel thread per evitare stalli. Questo si ottiene spesso tramite upcalls dal kernel alla libreria thread utente.

    Semantica di fork() e exec()

    Nei sistemi multithread, il comportamento delle system call per la creazione di processi subisce delle variazioni importanti:

    • fork(): I fork creano una copia di tutti i thread o solo del thread chiamante? I sistemi UNIX moderni offrono solitamente entrambe le versioni della fork() per supportare ciascuna modalità a seconda della necessità.
    • exec(): Questa system call solitamente rimpiazza l'intero processo in esecuzione, includendo tutti i suoi thread.

    L'uso combinato: se una exec() viene chiamata immediatamente dopo una fork(), duplicare tutti i thread risulta inutile, motivo per cui si preferisce usare la fork che duplica solo il thread chiamante.

    Gestione dei segnali

    I segnali sono usati (soprattutto nei sistemi UNIX) come software interrupt per notificare a un processo che un particolare evento è accaduto. Il ciclo di vita di un segnale prevede che:

    1. Il segnale venga generato da un particolare evento.
    2. Il segnale venga spedito a un processo.
    3. Il segnale venga gestito da un apposito gestore (Default o User-defined).
    Il dilemma Multithreading: Dove viene spedito il segnale in un sistema multi-thread? Ci sono 4 approcci possibili:
    • Al singolo thread a cui il segnale fa riferimento.
    • A tutti i thread del processo.
    • A certi thread specifici del processo.
    • Assegnare a un thread specifico (dedicato) la ricezione di tutti i segnali per l'intero processo.

    Cancellazione di un thread

    L'interruzione di un thread prima che abbia completato il suo compito è detta cancellazione. Il thread che deve essere cancellato prende il nome di target thread. Esistono 2 approcci principali:

    • Cancellazione Asincrona: Termina il target thread immediatamente.
    • Deferred Cancellation (Differita): Permette al target di controllare periodicamente (in punti sicuri) se deve essere cancellato.

    Cancellazione di un Pthread:

    Nel caso dei Pthread, la cancellazione invocata dal codice è solo una richiesta; l'effettiva terminazione dipende dallo stato del thread. Se la cancellation mode è disabilitata, la richiesta rimane in stallo fino a quando il thread non la abilita. Il tipo di cancellazione di default è la Deferred.

    Thread-Local Storage (TLS)

    Il Thread-Local Storage (TLS) è uno spazio di archiviazione che permette ad ogni thread di avere la propria copia unica e privata dei dati.

    • Risulta molto utile quando non abbiamo pieno controllo della creazione del thread (come ad esempio può verificarsi nella creazione di un thread implicito o nei Thread Pool).
    • Differenza dalle variabili locali: Le variabili locali sono visibili solo nella funzione chiamante in cui sono dichiarate; i dati TLS, invece, sono visibili attraverso tutte le chiamate a funzione all'interno di quello specifico thread.
    • Sono molto simili ai dati static globali, ma con la differenza che sono unici per ogni thread creato.

    Implementazioni OS: Linux e Windows

    Linux non fa alcuna distinzione architetturale tra processi e thread: entrambi sono visti come Task. La creazione avviene tramite la potente system call clone(). Il programmatore passa dei flag a clone() per decidere cosa condividere con il "figlio": se si condividono memoria, file e segnali, il task risultante è di fatto un thread; se non si condivide nulla, il comportamento è equivalente a una fork() tradizionale.

    Windows implementa nativamente il mapping One-to-One. La Windows API definisce i thread a livello kernel: ognuno contiene un Thread ID, i registri (context), uno stack per lo User Mode, uno per il Kernel Mode e il Thread-Local Storage per i dati privati.
