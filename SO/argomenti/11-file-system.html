    Concetti Base: File e Directory fondamento

    Un file è la più piccola unità di storage logico: una sequenza di byte con un nome, dei metadati (permessi, proprietario, dimensione, timestamp) e un contenuto. Il SO astrae i dettagli fisici del disco e presenta i file tramite operazioni uniformi: open(), read(), write(), close().

    Una directory è una struttura speciale che mappa nomi a file (o ad altre directory). In Unix, una directory è anch'essa un file, il cui contenuto è una tabella di coppie (nome inode number). Le strutture di directory possono essere: a singolo livello (tutti i file in un'unica directory — primitivo), a due livelli (una directory per utente), ad albero (la struttura gerarchica usata da tutti i SO moderni), o con link (grafo aciclico o generale, che permette a un file di apparire in più directory).

    Allocazione dei File su Disco

    Esistono tre strategie per allocare i blocchi fisici di un file. Allocazione contigua: i blocchi di ogni file sono fisicamente consecutivi. L'accesso sequenziale e casuale è velocissimo (l'HDD può fare prefetch), ma soffre di frammentazione esterna e richiede di conoscere la dimensione del file alla creazione. Usata su CD-ROM e alcuni filesystem ottimizzati per streaming.

    Allocazione linked (concatenata): ogni blocco contiene un puntatore al blocco successivo. Nessuna frammentazione esterna, facilmente espandibile, ma l'accesso casuale richiede di attraversare la catena dal blocco. La FAT è un'ottimizzazione di questa strategia che porta i puntatori fuori dai blocchi dati in una tabella separata.

    Allocazione indexed (inode su Unix): un blocco indice contiene i puntatori a tutti i blocchi dati del file. Accesso diretto efficiente per file di qualunque dimensione. Il costo è lo spazio del blocco indice, ma i puntatori indiretti (singolo, doppio, triplo) permettono di gestire file enormi con strutture a taglia variabile.

    inode (Unix/Linux)

    L'inode è la struttura dati centrale del filesystem Unix/Linux. Ogni file ha esattamente un inode, identificato da un numero univoco nel filesystem. L'inode contiene: permessi (rwx per owner/group/other), UID e GID del proprietario, dimensione del file, timestamp (accesso, modifica, cambio metadati), numero di hard link, e i puntatori ai blocchi dati.

    I puntatori sono organizzati in tre livelli: 12 puntatori diretti (puntano direttamente ai primi 12 blocchi dati — coprono file fino a 48 KB con blocchi da 4 KB), un puntatore indiretto singolo (punta a un blocco che contiene altri puntatori — aggiunge fino a 4 MB), un puntatore indiretto doppio (~4 GB aggiuntivi) e un puntatore indiretto triplo (~4 TB). Il nome del file non è nell'inode: è nella directory che lo contiene. Più nomi (hard link) possono puntare allo stesso inode.

    FAT — File Allocation Table

    La FAT è un filesystem semplice che risolve il problema dell'allocazione linked portando i puntatori fuori dai blocchi dati. Esiste una tabella (la FAT) con una entry per ogni cluster del disco. Ogni entry contiene il numero del cluster successivo nella catena del file, oppure un marcatore di fine file (EOF) o di cluster libero.

    Esistono tre varianti: FAT12 (floppy, partizioni piccole), FAT16 (partizioni fino a 2-4 GB), FAT32 (fino a 2 TB). La FAT è universale (supportata da praticamente tutti i SO), semplice da implementare, ma non ha journaling, permessi avanzati né supporto per file oltre 4 GB (FAT32). Ancora oggi è il filesystem standard per chiavette USB, schede SD e partizioni di boot EFI.

    Journaling

    Il problema del filesystem senza journaling: se il sistema si spegne improvvisamente durante un'operazione di scrittura (es. creazione di un file che richiede l'aggiornamento dell'inode, della directory e del bitmap dei blocchi liberi), il filesystem può restare in uno stato inconsistente. Il tool fsck può ripararlo, ma richiede una scansione completa del disco — lenta e non garantita.

    Il journaling registra le modifiche imminenti in un'area dedicata (il journal o log) prima di applicarle al filesystem. Se il sistema crasha, al riavvio basta ripercorrere il journal e rieseguire le operazioni incomplete: il recovery è veloce e garantisce la consistenza. Il prezzo è un overhead di scrittura (ogni operazione viene scritta due volte). Usato da ext4 (Linux), NTFS (Windows), APFS (macOS), XFS e quasi tutti i filesystem moderni.

    VFS — Virtual File System

    Il Virtual File System è uno strato di astrazione nel kernel che fornisce un'interfaccia uniforme sopra filesystem diversi. Un'applicazione che chiama open("/mnt/usb/file.txt") usa la stessa syscall indipendentemente dal fatto che /mnt/usb sia FAT32, ext4, NTFS, NFS o procfs.

    Il VFS definisce strutture dati astratte — inode, dentry (directory entry), file, superblock — e un insieme di operazioni virtuali (read(), write(), lookup()). Ogni filesystem registra le proprie implementazioni di queste operazioni. Quando il kernel deve leggere un file, invoca la funzione read() del filesystem corretto tramite un puntatore a funzione. I filesystem speciali come procfs (che espone informazioni sui processi come file in /proc) e sysfs utilizzano lo stesso meccanismo VFS.

    Mounting e Link

    Il mounting è l'operazione con cui un filesystem viene agganciato all'albero delle directory del SO in un mount point. Con mount /dev/sdb1 /mnt/dati, il contenuto del disco sdb1 diventa accessibile sotto /mnt/dati come se fosse parte del filesystem principale. Più filesystem (con tipi diversi) coesistono nell'unico albero delle directory.

    I hard link (ln src dst) creano un secondo nome nella directory che punta allo stesso inode. Non c'è copia dei dati; il file esiste finché almeno un hard link punta all'inode (link count > 0). I hard link non possono attraversare filesystem diversi e non possono puntare a directory. I soft link (symlink, ln -s src dst) sono file speciali che contengono un percorso. Possono attraversare filesystem, puntare a directory e puntare a target inesistenti — ma se il target viene cancellato il link diventa dangling.
