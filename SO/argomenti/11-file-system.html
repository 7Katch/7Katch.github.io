    Il Concetto di File

    Un File è un'astrazione dell'OS che definisce uno spazio di indirizzamento logico contiguo, mappato su settori fisici di un disco. L'estensione spesso identifica il tipo, ma in Unix non è obbligatoria (si usa un "magic number" interno). I file possiedono un Identifier (tag univoco interno), Location, Size, Date, e diritti di Protection.

    Record Logici e Fisici

    L'OS legge/scrive usando unità base minime dette Record Logici, ma l'Hardware legge interi Record Fisici (settori). Quasi mai combaciano perfettamente: ad esempio su 512 byte di settore fisico, se un file usa solo 10 byte, i restanti 502 byte del blocco vanno sprecati. Questo causa Frammentazione Interna.

    Open-File Table e Lock

    La system call open() carica le informazioni del file in memoria per non dover accedere al disco ogni volta che si invoca una read/write. Il kernel mantiene una System-wide Open-File Table e una tabella Per-Process (con il current file pointer, open count e diritti di accesso).

    Se più processi aprono lo stesso file, si possono richiedere dei Locks:

    • Shared Lock: Molti lettori concorrenti.
    • Exclusive Lock: Un solo scrittore per volta.
    • Mandatory: l'OS impedisce fisicamente ad altri di accedere. Advisory (Unix): l'OS dice ai processi che il file è lockato, ma sta a loro decidere di fermarsi o meno.

    Accesso e Directory

    L'accesso ai dati può essere: Sequenziale (letto dall'inizio alla fine, tipico dei nastri) o Diretto (Random) (si salta a un record specifico tramite un indice). I file sono organizzati in Directory.

    Struttura delle Directory

    Per nominare ed esplorare le directory in modo intelligente (senza conflitti di nome tra utenti) si usa un albero logico:

    • Tree-structured: Permette path Assoluti (da `/`) e Relativi (da `.`).
    • Acyclic-Graph: La struttura ad albero classica in realtà in Linux può avere sotto-alberi condivisi tramite Aliasing, diventando un grafo aciclico.
    • General Graph: Ammette cicli. Serve un Garbage Collector che periodiciamente cerchi e ripulisca le sezioni disco irraggiungibili.

    Esempio di Grafo Aciclico (Aliasing)

    flowchart TD root[/Root /] --> bin[/bin/] root --> dict[/dict/] root --> spell[/spell/] dict --> list[list] spell --> words[words] words -.->|Hard Link| list

    Il file "list" è condiviso. Due percorsi diversi portano allo stesso identico file fisico.

    Implementazione (Linear vs Hash)

    Sotto il cofano, le directory sono implementate come: Liste Lineari (semplici da programmare ma con tempi lineari lunghissimi di ricerca O(n)) o come Hash Tables (ricerca quasi istantanea, ma bisogna gestire le collisioni con un chain overflow).

    Links in Unix (ln vs ln -s):
    Hard Link: Crea un vero puntatore parallelo al file fisico (FCB/inode). Se elimini l'originale, il file vive finché il contatore di hard links non va a 0.
    Symbolic Link: È un "puntatore a un puntatore" (una stringa che contiene il path originale). Se cancelli l'originale, il symlink resta "orfano" (dangling pointer) e si rompe.

    Strati del File System

    Per garantire modularità, il sistema operativo divide il modulo File System in strati (Layers):

    1. Application Program
    2. Logical File System: Gestisce i metadati e la struttura a directory. Traduce i nomi in file ID / Inode Number (File Control Blocks - FCB). Non tocca i dati veri.
    3. File-organization Module: Converte il blocco logico (chiesto dall'utente) nel blocco fisico (settore del disco). Gestisce l'allocazione dello spazio.
    4. Basic File System: Interroga il controller HW: "leggi blocco fisico 123", gestendo RAM buffers e cache.
    5. I/O Control / Device Drivers: Invia bit e correnti elettriche all'hardware.

    Metodi di Allocazione

    Il File System deve mappare un file logico in blocchi di byte sparpagliati (o meno) sul disco fisico.

    1. Contiguous
    Ogni file occupa una serie ininterrotta di blocchi fisici continui.
    Performance immense (lettura con seek=0).
    Problema grave: impossibile trovare facilmente lunghi spazi vuoti (Frammentazione Esterna). Un file in crescita va per forza spostato e riallocato (Extents aiutano ma non risolvono).
    2. Linked
    I blocchi sono sparsi a caso ovunque. Ogni blocco contiene un puntatore (link) al blocco successivo.
    Nessuna frammentazione esterna. Si spreca lo spazio del puntatore (Framm. Interna).
    Pessimo per accesso diretto: per leggere l'n-esimo blocco, devi scorrere dall'inizio! Un'evoluzione è la FAT (File Allocation Table), che tiene i link in una tabella RAM velocissima.
    3. Indexed
    I blocchi sono sparsi, ma i loro puntatori sono radunati in un Unico Blocco Indice.
    Accesso diretto istantaneo! Ma per i file grossi un solo blocco indice non basta, servono indici di indici (multilevel).

    Combined Scheme (L'Inode Unix/Linux)

    Unix unisce i vantaggi dell'allocazione Indexed risolvendo il problema della taglia limite.
    L'Inode (Index Node) è l'FCB di Linux. Al suo interno ha, per esempio, 15 slot di puntamento:

    • Slot 1..12 (Direct): Puntano direttamente ai primi 12 blocchi dati (ideale per i 90% di micro-file).
    • Slot 13 (Single Indirect): Punta a un intero blocco che contiene solo altri puntatori a dati.
    • Slot 14 (Double Indirect): Punta a un blocco che punta a blocchi indice, che puntano a dati (enorme).
    • Slot 15 (Triple Indirect): Tre salti. Può mappare TeraBytes!

    Simulazione Interattiva: Allocation Methods

    Osserva come i file vengono disposti sui blocchi logici del disco a seconda del metodo scelto.

    Free-Space Management

    Per allocare nuovi blocchi, il sistema deve sapere cosa è vuoto. Per tracciare lo spazio libero, il SO può usare:

    • Bit Vector (Bitmap): 1 bit per ogni singolo blocco fisico del disco (0=occupato, 1=libero). Facilissimo e veloce se sta tutto in RAM. Su dischi immensi (1TB con 4KB block = bitmap da 32MB!) la RAM inizia a riempirsi troppo, si usano raggruppamenti.
    • Linked List (Free List): Ogni blocco vuoto punta al successivo blocco vuoto. Non si spreca memoria per la mappa, si prende in O(1) la "testa" della lista per allocare. Lentissimo per cercare spazi contigui.
    • Grouping & Counting: In un blocco vuoto salvo gli indirizzi di N altri blocchi vuoti. Oppure memorizzo `(indirizzo_inizio, conteggio_blocchi_liberi_contigui)`. Ottimo per extents e allocazione contigua.

    VFS e Sistemi Distribuiti (NFS)

    In un computer possono coesistere dozzine di partizioni logiche (volumi) e file system diversi (NTFS per Windows, ext4 per Linux, ISO 9660 per un DVD, chiavetta USB fat32). Vengono uniti al / logico in fase di Mounting.

    Come fa Linux a lanciare read() o open() senza impazzire su formati incompatibili? Usa il Virtual File System (VFS).

    Il VFS è un'interfaccia Orientata agli Oggetti fornita dal Kernel Linux. Le app chiamano la API del VFS. Il VFS guarda le sue tabelle dinamiche (struct file, inode, dentry, superblock) e inoltra la richiesta al driver specifico (ext4, FAT) che realizza concretamente la funzione.

    Network File System (NFS)

    Creata da Sun, NFS sfrutta proprio il livello VFS per montare (mount) una directory di un altro PC remoto come se fosse una cartella locale. Il VFS smista le operazioni di lettura "locali" all'hard disk, e le chiamate di lettura "NFS" allo stack di rete, inviando le richieste al server remoto tramite RPC (Remote Procedure Call) su protocolli XDR.

    Consistency Semantics

    Condividere file porta il problema della coerenza (quando gli altri utenti vedono i dati che io modifico?):

    • UFS (Unix File System): I dati modificati e scritti su un file aperto diventano immediatamente visibili a tutti gli altri utenti connessi. I puntatori I/O possono essere condivisi.
    • AFS (Andrew File System): Le modifiche ai dati rimangono private. Diventano visibili agli altri solo *dopo che la sessione è stata chiusa*.
