Sistemi Real-Time fondamento

Un sistema real-time è un sistema in cui la correttezza del risultato dipende non solo dal valore computato, ma anche dal momento in cui viene prodotto. Un risultato corretto arrivato troppo tardi è un risultato sbagliato. I task in un sistema RT hanno deadline: un tempo massimo entro cui devono completare la loro elaborazione.

I campi applicativi sono critici per la sicurezza umana: sistemi di controllo ABS, pacemaker, avionici, controllo di reattori nucleari, robot industriali, sistemi di guida autonoma. Un SO general-purpose come Linux non è adatto di default a questi scenari perché non garantisce latenze deterministiche: un interrupt può essere ritardato da decine di millisecondi a causa dello scheduler, della gestione degli interrupt stessi o del memory management.

Hard vs Soft Real-Time

I sistemi RT si dividono in due categorie in base alle conseguenze del mancato rispetto di una deadline. In un sistema Hard Real-Time, ogni deadline mancata è un fallimento catastrofico del sistema — potenzialmente con conseguenze fisiche gravi. Esempi: controllo di un airbag (deve attivarsi entro 30 ms dall'impatto), sistema di controllo di un reattore (deve rispondere alle anomalie in microsecondi), controllore di volo di un aereo. Non è accettabile nessuna tolleranza: la deadline è assoluta.

In un sistema Soft Real-Time, una deadline occasionalmente mancata degrada la qualità del servizio ma non è catastrofica. Esempi: streaming video (un frame in ritardo produce un artefatto visivo temporaneo), interfaccia utente (una risposta lenta è fastidiosa ma non pericolosa), trading ad alta frequenza (una transazione in ritardo perde opportunità ma non causa danni fisici). La maggior parte degli RTOS consumer-grade (Android RTOS, iOS) rientra in questa categoria.

Rate Monotonic Scheduling (RMS)

Il Rate Monotonic Scheduling è l'algoritmo di scheduling real-time più studiato, proposto da Liu & Layland nel 1973. È un algoritmo preemptive a priorità statica: le priorità dei task vengono assegnate prima dell'esecuzione e non cambiano mai. La regola di assegnamento è semplice e ottimale: il task con il periodo più breve riceve la priorità più alta.

RMS è ottimale tra tutti gli algoritmi a priorità statica: se un insieme di task periodici è schedulabile con un qualunque algoritmo a priorità fissa, lo è anche con RMS. Il suo limite teorico di utilizzazione è U = n × (2^(1/n) − 1), che converge a ~69.3% per n grande. Significa che se l'utilizzo totale della CPU è sotto il 69%, RMS garantisce il rispetto di tutte le deadline. Sopra il 69%, potrebbero verificarsi deadline mancate, anche se non è detto (il test è sufficiente ma non necessario).

EDF — Earliest Deadline First

L'Earliest Deadline First è un algoritmo di scheduling preemptive a priorità dinamica: in ogni istante, il task con la deadline più vicina nel tempo riceve la massima priorità. Le priorità cambiano dinamicamente al variare delle deadline dei task nella ready queue.

EDF è ottimale per sistemi monoprocessore: può schedulare qualsiasi insieme di task periodici schedulabile da qualunque altro algoritmo. Il suo limite di utilizzazione è il 100%: se la somma delle utilizzazioni (tempo di computazione / periodo) di tutti i task è ≤ 1, EDF garantisce il rispetto di tutte le deadline. Il prezzo è la complessità implementativa: le priorità devono essere ricalcolate dinamicamente, e la struttura dati della ready queue deve supportare aggiornamenti efficienti (tipicamente un heap). EDF non è deterministico in overload (utilization > 1): non è prevedibile quali task mancheranno la deadline.

Caratteristiche di un RTOS

Un Real-Time Operating System è progettato per garantire comportamento deterministico e latenze prevedibili. Le caratteristiche fondamentali che lo distinguono da un SO general-purpose sono: latenza di interrupt minima e deterministica (il tempo tra l'arrivo di un interrupt e l'esecuzione del suo handler deve essere garantito e limitato), preemption immediata (un task ad alta priorità deve poter preemptare uno a bassa priorità in qualsiasi momento, anche dentro una syscall), assenza di memoria virtuale (il demand paging introduce latenze non deterministiche per i page fault — i sistemi hard RT usano solo RAM fisica), clock ad alta risoluzione (timer hardware in grado di innescare interrupt con precisione di microsecondi o nanosecondi).

Esempi di RTOS diffusi: FreeRTOS (open source, usatissimo in microcontrollori, supporta ARM Cortex-M), VxWorks (commerciale, usato in avionica e militare), QNX (usato in automotive, sistemi medicali, infrastrutture critiche), Zephyr (open source Linux Foundation, per IoT e wearable).
